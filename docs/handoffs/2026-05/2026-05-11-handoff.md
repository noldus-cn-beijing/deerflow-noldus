### 关键工作机制

**{{handoff://}} 占位符流程**：
1. Lead 在 task() prompt 中使用 `{{handoff://code_executor}}` 等占位符
2. task_tool 的 `_resolve_handoff_placeholders()` 将其替换为 `/mnt/user-data/workspace/handoff_code_executor.json` 真实路径
3. 同时采集 `authorized_handoff_paths` set，透传给 SubagentExecutor
4. SubagentExecutor._create_agent() 将 HandoffIsolationProvider + GuardrailMiddleware append 到中间件链
5. Subagent read_file handoff_*.json 时，GuardrailMiddleware 调用 HandoffIsolationProvider.evaluate() 检查授权

**HandoffIsolationProvider 授权逻辑**：
- `agent_id` 为空 → lead 调用，直接放行
- `tool_name != "read_file"` → 非 read_file 工具，直接放行
- 文件路径不含 `handoff_` 或不以 `.json` 结尾 → 非 handoff 文件，直接放行
- 文件路径匹配自己的 outbox（如 data-analyst 读 handoff_data_analyst.json）→ 放行
- 文件路径在 `authorized_paths` 中 → 放行
- 否则 → deny（code: `handoff_isolation.unauthorized`）

**[gate_signals] 生成路径**：
- **code-executor**：胶水脚本（analysis.py）由 ethoinsight-code skill 模板生成，print [gate_signals] 到 stdout，模型转述给 lead
- **data-analyst**：模型按 system_prompt 中的 `<gate_signals_contract>` 直接生成
- **report-writer**：模型按 system_prompt 中的 `<gate_signals_contract>` 直接生成

## 4. 关键发现

1. **GuardrailMiddleware 不自动填充 is_subagent**：`_build_request()` 使用 `agent_id=self.passport` 填充身份，`is_subagent` 默认为 False。HandoffIsolationProvider 改用 `agent_id` 是否为空来判断 subagent/lead 身份。实例化时传入 `passport=f"subagent:{config.name}"`。

2. **code_executor 的最终消息来自胶水脚本 stdout**：仅改 system_prompt 不够，必须改 ethoinsight-code skill 的范式模板（Task 7a 做了这个）。system_prompt 改动的作用是告诉模型"保留 stdout 中的 [gate_signals] 块不要删"。

3. **prompt.py 使用 f-string**：`_build_subagent_section()` 返回 f-string（`return f"""<subagent_system>`）。在 f-string 中，`{{{{handoff://...}}}}` → 输出 `{{handoff://...}}` → task_tool 解析为真实路径。

5. **config.yaml 残留清理**：启动 dev 环境时遇到 `ImportError: ethoinsight.templates.tool`。原因是 Phase 4 删除了 6 个旧 langchain 工具（`tool.py`），但 `config.yaml`（本地文件，不在 git）仍引用它们。已修复：删除了 `ethoinsight` tool group 和 6 个工具定义。不影响功能——这些工具已被胶水脚本模式替代。
