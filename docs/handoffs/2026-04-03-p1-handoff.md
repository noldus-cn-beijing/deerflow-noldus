### 3.2 ethoinsight 与 DeerFlow 的集成方式

**uv workspace 方案**（非 pip install -e）：
- `backend/packages/ethoinsight` 是一个 symlink → `../../../../packages/ethoinsight`
- `backend/pyproject.toml` 中 `[tool.uv.workspace].members` 包含 `"packages/ethoinsight"`
- `backend/pyproject.toml` 中 `[tool.uv.sources].ethoinsight = { workspace = true }`
- `uv sync` 自动安装为 editable 模式

### 3.3 EthoVision 数据格式（P1 实测确认）

**文件类型**：
- `轨迹-*.txt` — 轨迹数据（有 `"标题行数："` header，UTF-16 LE + 分号）
- `统计数据-*.txt` — 统计汇总（无 header 行数声明，第一行直接是列名）
- `.xlsx` — Excel 统计汇总（斑马鱼、足迹分析）
- `.csv` — UTF-8（仅新奇抑制摄食）

**轨迹文件 header 结构**：
- 第 1 行声明 header 行数（37-41 行不等，非固定 39）
- 倒数第 2 行（相对 header_lines）= 列名
- 倒数第 1 行 = 单位
- 之后 = 数据行

**列名差异大**：
- 不同范式列名完全不同
- 同一范式不同导出配置列名也不同（如 EPM 4 个文件有 4 套不同列名）
- 复杂列名如 `Nose within object zone(Familiar object 1 /鼻尖)` 需动态处理
- `utils.py` 中的 `normalize_column_name` 处理了精确匹配、正则模式、fallback slugify 三层

**Demo 数据位置**：`/home/qiuyangwang/noldus-insight/demo-data/DemoData/`
- 16 个范式目录
- 斑马鱼最完整：35 个轨迹文件
- 旷场/强迫游泳只有统计文件，无轨迹文件

## 4. 关键决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| ethoinsight 集成方式 | uv workspace（方案 A） | 与 deerflow-harness 一致，`uv sync` 自动安装，最干净 |
| config.yaml API 配置 | NewAPI 中转 `https://newapi.noldusapi.com/v1` | 用户使用 NewAPI 中转智谱 API |
| build-backend | hatchling（从 setuptools 改） | uv workspace 要求一致的 build system |
| 列名标准化策略 | 精确匹配 → 正则模式 → fallback slugify | 列名差异太大，无法穷举，需要 fallback |
| parse_batch 过滤 | 只解析含 `"标题行数"` 的轨迹文件 | 统计文件格式完全不同，需要单独处理 |

## 7. 风险与注意事项

- **统计文件未解析**：当前 parse_batch 只处理轨迹文件。旷场/强迫游泳 demo 只有统计文件，P2 可能需要加统计文件解析
