# EthoInsight LLM 微调策略

> 日期: 2026-04-09
> 状态: 规划阶段

---

## 1. 背景与动机

当前 EthoInsight agent 使用 GLM-5.1 API，存在两个核心问题：

1. **指令遵循差** — 不按 checklist 执行、多步推理容易跑偏，需要大量 prompt 工程约束
2. **行为学领域知识缺失** — 模型不理解范式和指标含义，无法做出合理的分析决策

领域知识部分已通过 noldus-kb MCP 知识库补充，但模型本身不内化这些知识，且指令遵循问题无法通过知识库解决。

**长期目标**: 拥有一个内化了行为学知识和 Noldus 产品知识、专门针对行为学数据分析微调的 LLM，作为 EthoInsight agent 的核心模型。

---

## 2. 总体路线：两步走

### 第一步：继续预训练（CPT）— 知识注入

**目标**: 让模型内化行为学领域知识和 Noldus 产品知识。

**训练数据**（已有）:
- Noldus 产品相关 publications
- 产品手册
- 实验范式文档

**特点**: 不需要 agent 交互数据，用已有的知识文档即可启动。

### 第二步：指令微调（SFT）— Agent 能力训练

**目标**: 让模型学会正确使用 tools、调度 subagent、处理多轮交互。

**训练数据**（需积累）:
- 真实用户与 agent 的交互记录
- 正确的 tool call 示例（输入→输出对）
- 好/坏案例标注（正确的分析流程 vs 错误的）

**前置条件**:
- Agent 架构稳定（当前正在重设计）
- 积累几百条真实交互数据
- 数据经过标注和清洗

---

## 3. 基座模型选型

### 硬性约束

| 约束 | 要求 |
|------|------|
| 部署硬件 | NVIDIA RTX 5090 (32GB VRAM)，需支持本地部署 |
| 参数量 | 7B-14B（FP16 推理下 32GB 显存的上限约 14B） |
| 语言能力 | 中英双语，英文行为学文献 + 中文用户交互 |
| Agent 能力 | Tool use / function calling 必须强，支持 subagent 调度 |
| 质量 vs 速度 | 质量优先 |
| 微调支持 | Fireworks.ai 托管微调服务 |
| 许可证 | 商业友好 |

### 候选模型对比

| | Qwen2.5-14B | Qwen3-8B | DeepSeek-R1-Distill-14B |
|--|--|--|--|
| 参数量 | 14B | 8B | 14B |
| 5090 FP16 | 刚好 | 宽裕 | 刚好 |
| 中英双语 | 强 | 强 | 强 |
| Tool use | 原生支持，社区验证多 | 原生支持，较新 | 偏推理链，tool use 需额外适配 |
| Agent 场景 | 大量实践案例 | 较新，案例少 | 擅长推理，agent 适配成本高 |
| 微调生态 | 最成熟 | 较新 | 一般 |
| 许可证 | Apache 2.0 | Apache 2.0 | MIT |

### 推荐：Qwen2.5-14B

理由：
- 14B 在 5090 上可 FP16 推理，质量优于 7B/8B 级别
- Tool use 能力在开源模型中最成熟，社区有大量 agent 场景验证
- 微调生态最好，Fireworks.ai 直接支持
- 双语能力强，覆盖行为学英文文献和中文用户交互场景

---

## 4. 微调平台

**选择: Fireworks.ai**

- 托管微调服务，无需自建训练基础设施
- 支持 CPT 和 SFT
- 微调完成后可直接通过 API 调用，也可导出模型权重用于本地部署

---

## 5. 执行计划

### 阶段 0：过渡期（现在）

- 先换用更强的基座模型 API（如 Qwen2.5 或 DeepSeek-V3）替代 GLM-5.1，让 agent 先跑通
- 同时推进 code-executor 架构重设计（降低对模型多步推理能力的依赖）

### 阶段 1：准备 CPT 数据

- 收集和清洗 Noldus publications、产品手册、范式文档
- 转换为适合 CPT 的训练语料格式
- 评估数据量是否充足

### 阶段 2：CPT 知识注入

- 在 Fireworks.ai 上对 Qwen2.5-14B 做继续预训练
- 验证模型对行为学概念的理解（人工评估 + 自动化测试集）
- 替换过渡期模型

### 阶段 3：积累 SFT 数据

- Agent 架构稳定后，收集真实用户交互记录
- 标注正确/错误的 tool call、分析流程
- 构建评估基准（benchmark）

### 阶段 4：SFT Agent 能力微调

- 在 CPT 后的模型基础上做 SFT
- 训练 tool use、subagent 调度、错误处理等 agent 能力
- 迭代评估和改进

---

## 6. 风险与注意事项

1. **架构未稳定时不做 SFT** — agent 的 tool 接口和 prompt 结构还在变，过早微调会浪费。
2. **CPT 数据质量** — publications 质量高但格式多样（PDF、HTML），需要投入清洗工作。
3. **14B 模型在 5090 上的实际表现** — 需要实测推理速度和显存占用，确认 FP16 可行，否则需考虑 INT8 量化或降到 8B。
4. **微调后的模型退化** — CPT/SFT 可能导致通用能力下降，需要在训练中混入通用数据。
5. **Fireworks.ai 平台依赖** — 如平台政策变化，需要有本地微调的备选方案（如 LLaMA-Factory）。
6. **评估体系** — 需要建立行为学领域的评估基准，否则无法衡量微调效果。
