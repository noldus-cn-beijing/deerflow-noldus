01
Why · 为什么需要方法论

AI 带来了能力，也带来了新的失控风险

🌀
过度自作主张
AI 补全了你没要求的"优化"，加了你不需要的抽象，改了你没注意到的边界条件。

结果：diff 变大、审查困难、引入隐患
🪤
静默失败 / 哑故障
AI 给出了看起来正确的代码，测试全绿，但实际上走了错误路径——错误被兜底机制悄悄藏起来。

结果：空心绿比诚实 FAILED 更危险
📡
上下文丢失与误判
AI 不了解项目历史，容易把表象当根因，把巧合当规律，把临时方案当永久设计。

结果：重复踩坑，叠加兜底，越修越脆
💡
AI 的价值在于执行力——它能快速写代码、生成测试、重构结构。
但判断力——什么该做、什么不该做、根因是什么——必须由人来掌控。
方法论的核心，就是让 AI 在正确的边界内发挥最大价值。
02
Principle 1 · 规格先行

Spec-First：先对齐，再执行

为什么 AI 更需要规格文档
AI 执行快，所以"跑题"的代价更高
人工编码跑偏，还在思考过程中能察觉。AI 几秒钟已经写完了五个文件——方向错了，撤销成本极高。
Spec 是人机之间的契约
明确写出：修改哪些文件、预期行为、不能动什么、验收标准是什么。AI 只做 Spec 说的事，不多也不少。
先写 Red Test，再让 AI 实现
Red Anchor Test = 先写一个必须失败的测试，然后让 AI 把它变绿。
防止 AI "实现了一个没有行为的 pass"。
Spec 不做，AI 会自己做决策
没有 Spec，AI 会做"合理推断"——它的推断不一定和你的业务背景一致。
一份 Spec 应该包含
# spec: fix-handoff-status-field.md ## 根因 handoff 字段三分裂：metrics / metrics_results / metrics_summary 校验器只认 metrics_summary → 28/56 条完整 handoff 被误判 ## 修改范围 executor.py L104 # 校验逻辑 schemas.py L78 # 字段定义 ## 不能动 - 上游 DeerFlow 相关文件 - 其他 handoff 类型 ## 验收标准 pytest test_handoff_field.py → GREEN dogfood FST 完整分析 → completed
⚠️
反模式："帮我修一下这个 bug"
→ AI 会猜测根因，可能修错位置，可能引入新问题
03
Principle 2 · 测试优先

TDD × AI：让测试成为合同，而不是补丁

AI 时代 TDD 的新意义
测试是 AI 行为的边界围栏
AI 生成代码时会持续跑测试自检。测试写得越精确，AI 越不会跑偏。
合并前必须全量通过
只跑新增测试是常见陷阱——修改了共享逻辑但没发现旧测试被悄悄破坏。
警惕测试隔离污染
有些测试单独跑全绿、全量跑必红——这是测试卫生债，不是你的代码写坏了，要区分清楚。
conftest mock 会掩盖真实问题
测试框架的 mock 可能让循环依赖、启动错误在 pytest 里假绿——
改核心模块后必须额外做裸导入验证。
三类关键测试
🔴 Red Anchor Test
先写必须失败的测试防止 AI 偷懒
🔁 等价性测试
重构前后行为一致必须看 baseline 内容
🚪 裸导入测试
绕过 conftest mock发现循环依赖
# ✅ 裸导入：绕过 conftest mock python -c "import app.gateway" → OK（无循环依赖） # ⚠️ 等价性测试必须看 baseline 内容 # 如果 6 个 baseline 全是同一句错误串 # → 调用签名写错被 except 吞了 → 假绿

Principle 3 · 诊断优先

根因未隔离前，不加兜底机制

AI 时代的误诊模式
同一表象，异质根因
同样的"任务卡死"现象可能源于：
① Prompt 矛盾造成叙述黑洞
② Schema 缺少合法状态值
③ 参数类型不匹配（标量 vs 列表）
每次都要实验确认，不能复用上次的结论。
决定性实验条件必须显式核实
"加固后仍失败"可能是在旧条件下测的——只能证明旧条件压不过，无法证明新根因不存在。
加兜底 = 给下次误判铺路
根因未隔离就加兜底机制，下次同类故障更难排查——兜底把错误藏进了"管线全绿"。
🚨
原则：先做决定性实验，隔离根因；
然后修根因；最后按需补防御层。
顺序不能倒。
正确的诊断流程
🔬
Step 1 · 复现
最小可复现场景，固定变量
📊
Step 2 · 读原始证据
看真实日志/headers，不靠猜测
🎯
Step 3 · 隔离根因
决定性实验：证伪竞争假设
🔧
Step 4 · 修根因
最小改动，不顺带"清理"
🛡️
Step 5 · 按需补防御
防御层可观测，有触发率指标
💡
实例：Cookie 403 故障——第一反应是"上游禁用了 Auth"，但 curl -D - 看 Set-Cookie 的 Max-Age 就发现：是 session cookie 生命周期不对称，不是 Auth 配置问题。

Principle 4 · 知识管理

单一事实来源 （SSOT）：知识只存一份

AI 协作中的知识双存危机
AI 文档 vs 代码：谁说了算？
AI 会同时读 Skill 文档、System Prompt、代码注释。三处描述不一致时，AI 自己选一个——通常选错的那个。
Skill 文档描述工具输出 → AI 脑补
Skill 写了"调用此工具会返回 X 格式的结果"→ AI 不再真正调用工具，直接伪造返回值。
规则：工具文档只教"何时调"，绝不描述"返回什么"。
Deny 消息必须含明确指令
只写"不允许此操作"不够——AI 知道错了但不知道该怎么办，会继续。
必须写："请改调 X 工具，原因是 Y，然后做 Z"。
SSOT 的实践规则
✅ 业务规则
只在代码里文档引用代码
✅ Schema 定义
单一文件其他地方 import
❌ 双存知识
文档 + 代码各一份必然漂移
❌ AI Prompt 里内嵌规则
无法通过测试验证难以追踪变更
💡
判断是否双存的标准：人能手改且手改能静默存活——如果你改了 A 但 B 自动跟着变（如生成代码），B 是物化视图，不是双存。

Principle 5 · 可观测性

兜底要可见：静默成功是最危险的

兜底机制的双刃剑
兜底是合理的工程手段
在边界条件下自动恢复，防止整个流程因单点失败而崩溃。这是好的。
静默兜底 = 把回归藏进绿色
兜底机制本身没坏过——坏的是它工作得太好，把上游回归藏在"管线全绿"下。兜底触发率悄悄升高，没有人知道。
空心绿 vs 诚实 FAILED
一个任务"完成"了，但关键字段是兜底填充的，数据不可信——这比报错更糟，因为下游会信任这个结果继续处理。
可观测性设计清单
来源标记
sealed_by: "auto" vs "agent"区分正常与兜底
触发率指标
写进验收标准不是日志备注
完整性对账
计划产出 vs 实际产出缺失就报错
❌ 反模式
兜底无日志触发率不监控下游不区分来源
⚠️
验收标准的写法：
"auto-seal 触发率 < 5%"——不是"日志里有 auto-seal 字样"
前者是指标，后者是摆设。
07
Principle 6 · 上下文工程

AI 没有记忆——上下文工程决定质量下限

📋
CLAUDE.md / 项目文档
把项目级约束写进持久化文件：
• 不能动哪些文件
• 受保护的设计决策
• 已知坑与解决方法

每次会话 AI 都会读取，无需重复解释
🤝
Handoff 文档
跨会话传递上下文：
• 已验证的事实（非假设）
• 当前阻塞原因
• 已完成的 artifacts
• 下一步明确指令

接手前必须核实声称的事实
🧠
记忆系统
把跨会话有价值的判断持久化：
• 用户偏好与工作方式
• 项目特有约定
• 已学到的教训

不应存：代码结构、git 历史（从代码读）
🚨
Handoff 接手陷阱：不能直接信任 Handoff 文档的声称——
先 grep 代码 + 解码真实 trace，确认声称的根因真实存在，再继续。
"前任说 X 是根因"≠ "X 确实是根因"
⚠️
上下文污染：AI 会把对话历史中的错误假设当作事实。
当你发现 AI 在重复一个错误结论时，明确纠正并让它重新推理，而不是在错误基础上继续叠加。

Principle 7 · Agent 工程

多 Agent 协作：边界清晰是一切的前提

Agent 边界设计原则
每个 Agent 只做一件事
职责越模糊，边界越容易被 AI 自作主张突破。明确声明"哪些工具可用，哪些绝对不能用"。
Handoff 协议是硬契约
Agent 之间通过结构化 JSON 传递状态，字段有 Schema 校验，不用自然语言传递关键信息。
Guardrail 放在工具调用前
不要靠 prompt 说"不要做 X"——在 Agent 调用工具之前，用硬代码检查并拒绝越界调用。
Lead Agent 不亲自执行
编排者只做派遣和协调，不自己写脚本、调分析库——越界执行是角色混乱的根源。
Middleware 模式：行为可组合
数据隔离层
线程独立工作区沙箱执行
授权控制层
Guardrail 工具授权行为白名单
上下文管理层
上下文压缩记忆更新
质量保障层
任务状态追踪Handoff 校验
数据采集层
训练数据收集行为日志
💡
Middleware 模式的优势：每个行为是独立模块，可以单独测试、开关切换、顺序调整，不修改核心逻辑。

Principle 8 · 代码管理

Fork 维护：定制与上游的长期共存策略

分层保护策略
默认全量跟随上游
上游的 bug 修复和性能提升应默认合入——主动跳过 = 主动制造分叉，维护成本指数级增长。
定制文件清单 + Surgical Merge
维护一份受保护文件列表，这些文件绝不整文件覆盖——逐行对比，只合并上游的非冲突改动。
清单本身也会过时
每次同步前，diff 所有触及文件——不能只查静态保护清单，新定制文件可能还没有被加入清单。
Prompt 文件合并的特殊规则
# ❌ 危险：只用 grep 确认关键字还在 grep "topOfMind" prompt.py Found! ← 关键字在，但约束可能已被削弱 # 真实案例： # BASE（强约束）： "MUST NOT be written into any memory field" # 合并后退化为（弱约束）： "Do not record uploads" # grep 串在，但枚举范围缩小了 # ✅ 正确：三方字节对账 diff <(sed -n '100,200p' base/prompt.py) \ <(sed -n '100,200p' merged/prompt.py) # 看：强句计数 / 枚举完整性 / 缩进层级
⚠️
关键词还在 ≠ 约束完整
强约束伞句可能被降格为弱约束子句，或者枚举范围被缩小——必须看强句计数和完整性，不只看关键词。

全量推送
AI 生成代码的特殊 CI 要求
阻塞 IO 检测
AI 生成的异步代码容易误用同步 IO——专项 gate 检测 async handler 里的阻塞调用。
导入循环检测
AI 跨文件写代码容易引入循环依赖——CI 里跑裸导入（非 pytest），绕开 mock。
Schema 兼容性验证
AI 修改 Schema 时可能不知道所有下游消费者——自动验证所有已知调用场景。
部署常见陷阱
部署不等于数据库迁移
docker compose up 不会自动给已存在的表加列——加列的变更必须单独跑迁移脚本。
漏推镜像 = 静默跑旧代码
多服务部署时漏推某个镜像，compose 不报错，直接用旧镜像启动——bug 没修好但看起来部署成功了。
dev 配置必须对齐到 compose
本地 make dev 的 flags 是产品属性——必须同步到 docker-compose.yaml，不能只放 .env。

Summary · 实践总结

AI 协同开发的九条核心原则

① Spec-First
先写规格文档，明确边界和验收标准，再让 AI 执行。避免 AI 自作主张跑偏。
② TDD 是护栏
先写 Red Test，再让 AI 实现变绿。测试是 AI 行为的边界约束，不是事后补丁。
③ 根因隔离优先
症状相同不等于根因相同。先做决定性实验，再修，最后按需加防御。顺序不能倒。
④ SSOT 原则
知识只存一份。禁止在 AI Prompt / 文档 / 代码三处各自维护同一个业务规则。
⑤ 兜底要可见
兜底触发率写进验收标准。静默成功比诚实 FAILED 更危险——空心绿是最大的风险。
⑥ 上下文工程
CLAUDE.md 存项目约束，Handoff 传跨会话状态，记忆系统存教训——三层互补。
⑦ Agent 边界清晰
每个 Agent 职责单一，Guardrail 硬约束，Handoff 有 Schema——不靠 Prompt 维持秩序。
⑧ Fork 维护策略
默认全量跟随上游，只对定制文件 Surgical Merge。清单本身也需要定期审查更新。
⑨ 严格走 CI/CD
AI 生成的代码同样需要全量测试、导入检测、人工 Review——不因为"是 AI 写的"就降低标准。
