数据分析流水线

以 Shoaling（群体聚集行为）范式为例，展示从上传数据到产出报告的完整过程。

1
研究员上传数据 + 描述实验
上传 5 个 EthoVision XT 导出的轨迹文件，输入"分析 Shoaling 行为，对照组 vs 实验组"
2
Lead Agent 规划任务
识别实验设计：paradigm=shoaling, groups=[control, treatment]，制定三步分析计划，串行分派子代理
3
code-executor 执行分析
调用 run_paradigm_analysis → 自动解析数据 → 正态性检验 → 选择统计方法 (Mann-Whitney U) → 计算效应量 (Cohen's d) → 生成 Raincloud / Violin / Trajectory 等图表
输出：code_summary.json（含全部指标、统计结果、图表路径）
4
data-analyst 专业解读
读取统计结果 → 查询 Noldus 知识库获取 Shoaling 范式正常范围 → 判断组间差异的生物学意义 → 识别大效应量 (d > 0.5) 的关键指标
输出：analysis_summary.md（含专业洞察）
5
report-writer 撰写报告
整合数据结果 + 专业解读 + 文献引用 → 撰写 APA 格式科学报告（Results: 统计数值 + 效应量 + 图表引用；Discussion: 解读 + 文献对比 + 局限性）
输出：report.md + 全部图表文件

能力边界与人工复核规则

我们不声称 AI "万能"。下表明确列出每一类分析任务的自动化程度、适用边界、以及何时必须人工复核。边界清晰，是可信度的前提。

能力 自动化 适用边界（代码确定性保证） 边界外的处理与人工复核
正态性判断
test_normality		 全自动 Shapiro-Wilk，n ≥ 3；返回 is_normal、p_value、W 全部可审计 n < 3：跳过检验，默认走非参数路径，并在报告打 data_quality_warnings.severity=warning
两组比较检验
compare_two_groups		 全自动 决策树：两组正态 → Levene → 等方差 t / Welch t；
非正态 → Mann-Whitney U；
配对 → paired-t 或 Wilcoxon 		n₁+n₂ < 6：自动标注"统计效力严重受限，II 型错误风险高"；
效应量 |d| > 0.5 但 p > α：自动标注"可能真实效应，建议扩大样本"
多组比较
compare_multiple_groups		 全自动 全组正态 + 方差齐 → One-way ANOVA + Tukey HSD；否则 → Kruskal-Wallis + Dunn 事后检验 组数 > 5 或含有序组别（剂量梯度）：提示考虑趋势检验，需研究员确认设计
多重检验校正 全自动 指标数 ≥ 2 时自动 Bonferroni 校正，α 按指标数平分 高相关性指标（如 distance_moved / velocity）：Bonferroni 偏保守，在报告中说明该局限性
组指标 vs 个体指标
group_level_metrics		 全自动 IID、polarity 等群体级时序指标**只保留在群体层**，不广播到 per_subject（杜绝"零方差伪数据"） 单鱼文件：group_level_metrics.applicable=False，报告中明确说明
领域阈值判断
assess.py		 AI 辅助 已知范式（Shoaling / EPM / OFT 等）的参考范围由 YAML 知识库定义，确定性比对 新范式或超出阈值表：由 data-analyst 基于 MCP 文献检索给出参考，**报告中明确标注"基于文献推断，非库内固化阈值"**
异常个体识别 AI 辅助 Z-score > 2 或 IQR 外的个体自动标记 排除决定：系统**不自动排除**，只提示研究员人工确认。排除前后的结果都保留在报告中
生物学解读 AI 辅助 基于统计结果 + 领域知识库生成自然语言洞察；每条洞察附引用锚点 **报告定位为"研究员的草稿助手，非论文最终版"**，所有解读需研究员复核后方可投稿
实验设计评价 人工为主 系统仅报告样本量、平衡性、缺失率等客观指标 实验设计是否合理、是否有混杂因素：系统提示常见风险清单，最终判断由研究员承担
因果推断 人工为主 系统只做关联性分析（组间差异） 因果结论（"药物 X 导致行为 Y"）**不在 AI 输出范围**，需研究员结合实验设计独立判断

核心原则：确定性规则能处理的交给 Python 代码（100% 可审计、可复现）；边界外的情况 AI 辅助生成参考意见，但在报告中标注来源与不确定性；涉及科学判断的环节（因果、投稿结论）由研究员最终拍板。

中间产物契约（Schema + 版本化）

子代理之间通过结构化 JSON / Markdown 文件在共享工作区传递数据。**所有 handoff 文件都有 Pydantic schema 契约**，lead agent 在读入时强制校验，契约失败不静默吞掉，直接走澄清流程。

📄 code_summary.json · CodeExecutorHandoff
code-executor 产出。lead agent 用 Pydantic 校验后传给 data-analyst。
{
  "status": "completed | partial | failed",      // 枚举约束
  "summary": "string",                             // 一句话状态
  "output_files": {                                // 分类文件清单
    "metrics": "metrics.csv",
    "statistics": "statistics.json",
    "charts": ["boxplot_distance.png", ...]
  },
  "metrics_summary": {                             // group → metric → stats
    "control":   { "mean_nnd": { "mean": 37.97, "std": 2.67, "n": 2 } },
    "treatment": { "mean_nnd": { "mean": 48.16, "std": 18.95, "n": 3 } }
  },
  "statistics": { ... },                           // 检验结果（test_used/p/效应量）
  "data_quality_warnings": [                       // 严重度分级
    { "severity": "warning", "metric": "all", "message": "n<3 无法做正态性检验" }
  ],
  "confidence": 0.0 ~ 1.0                          // 可选整体置信度
}
版本管理：extra="allow" 允许向前兼容新字段；破坏性变更需升级 schema 主版本（例如字段重命名）。 依赖关系：data-analyst 和 report-writer 都从该文件读取，不可跳过生成。
📄 analysis_summary.md + handoff JSON · DataAnalystHandoff
data-analyst 产出。Markdown 给人读，结构化 JSON 给 lead 渲染洞察用。
{
  "status": "completed | failed",
  "analysis_summary_path": "analysis_summary.md",  // 人读版
  "key_findings": ["<=5 bullet 关键发现"],
  "excluded_metrics": [                            // 被剔除的指标及理由
    { "metric": "mean_iid", "reason": "单鱼样本不适用群体指标" }
  ],
  "references": [                                  // 引用锚点
    { "claim": "NND > 50mm 提示凝聚性下降",
      "source": "noldus-kb://shoaling/nnd-reference",
      "citation_key": "Miller2017" }
  ],
  "confidence_per_finding": { ... }                // 每条洞察的置信度
}
上游：code_summary.json。下游：report-writer 读取后写入 Discussion 段。引用锚点：references 直接指向知识库条目或文献 citation_key，可被前端渲染为脚注。
📄 report.md · APA 格式报告
report-writer 最终产出。结构固定：Methods / Results / Discussion / References。
约束：每个统计陈述必须附 test_used, statistic, p, effect_size；每条解读必须有 [citation_key] 引用锚点；无法引用的推论以"探索性假设"明确标记。 上游：code_summary.json + analysis_summary.md。版本策略：输出带时间戳文件名 report_YYYYMMDD_HHMMSS.md，历次运行可审计对比。

流程稳定性保障

Agent 流水线跑 3~5 分钟，中间每一步都有可能失败。我们通过 LangGraph 的中间件链 + 持久化 checkpoint 保证失败不丢数据、可恢复、可追溯。

⚙️ 阶段状态流转
每个 subagent 任务有显式状态：pending → running → completed | failed。lead agent 读取状态后决定是否推进或澄清。**绝不静默 bypass 失败**。
🔁 失败重试与澄清
LLM 429 / 网络错误：llm_error_handling_middleware 5s/15s/30s 三级退避；
工具失败：tool_error_handling_middleware 捕获 + 结构化错误反馈给 lead；
业务失败（如数据不符预期）：统一走 ask_clarification，**不自动重试同一 subagent 避免循环**
💾 阶段恢复（Checkpoint）
AsyncSqliteSaver 持久化每次 LLM 调用后的 state；前端意外断线后可从最后 checkpoint 继续，不重跑已完成阶段；archiving_summarization 压缩长对话时同步归档到 archived_messages/ 便于前端历史恢复
🛡️ 循环检测
loop_detection_middleware 用稳定 key hash 检测连续重复调用；subagent_limit_middleware 限制单 subagent max_turns（code-executor=12, data-analyst=6）；超限自动 abort 并报告
🏠 并发隔离（共享工作区）
每个会话 thread_id 对应独立沙箱目录 {thread_dir}/user-data/workspace/；虚拟路径 /mnt/user-data/workspace/ 映射到物理隔离目录，**不同会话零互相污染**；sandbox_audit_middleware 审计所有 shell 命令
📦 write_file 鲁棒性
显式 Pydantic _WriteFileArgs schema + 8000 字符阈值 fail-fast；超长内容自动指引 append 分段，避免 LLM 截断导致静默写入残缺文件

知识检索与证据链设计

"AI 写的报告引用从哪来？靠不靠谱？" —— 这是可信度的核心问题。我们用**三层证据链**确保每条引用都可追溯。

层级 来源 检索与过滤规则 在报告中的表现
L1 · 代码级事实 ethoinsight 库直接计算的统计量、效应量、样本量 **无 LLM 参与**，直接从 statistics.json 读数 "对照组 mean_nnd = 37.97 ± 2.67 (n=2)" · 可点击跳转到原始数据文件
L2 · 库内固化知识 Skills YAML 里的范式参考范围、阈值表、标准方法论 精确匹配：按 paradigm + metric 查表；命中则必引 "NND 正常范围 30~60 mm（基于 Noldus 行为学库）" · 引用锚点 skill://ethoinsight-analysis#shoaling-nnd
L3 · MCP 文献检索 noldus-kb 6200+ 同行评审论文（v0.1 启用） 向量检索 + 元数据过滤（期刊、年份、物种、范式）；
**只保留 score > 阈值且元数据匹配的结果**；
每条结果附 DOI + 摘要 + 相似度 		每条 Discussion 陈述必须有 [citation_key]，点击跳转到原文链接 + 摘要预览

引用锚点与审核机制

🔗 引用锚点（Citation Anchor）
每条 AI 生成的解读都有 {claim, source, citation_key} 三元组。source 取值：code://（代码计算）/ skill://（库内知识）/ noldus-kb://（MCP 文献）。无法归因的陈述自动标记为"探索性推论（无直接文献支持）"
🧪 事实一致性检查（v0.1 后）
report-writer 输出后，独立 verifier 对比报告中的数值 vs statistics.json，任何不一致（如 p 值写错、效应量虚构）直接拒收重写。**不允许 LLM 自由发挥数字**
📋 统计决策依据展示
报告 Methods 段自动生成："采用 X 检验，原因：Shapiro-Wilk 检验 p=0.03 拒绝正态性，因此选择非参数 Mann-Whitney U"。**决策路径完全可审计**，研究员能看到每一步判断的依据
👥 人工复核标记
报告最终段自动附"AI 辅助声明"：列出哪些结论是代码事实、哪些是库内知识、哪些是 AI 推断；研究员在投稿前可逐条确认或修改

关键设计决策

为什么用 4 个专业子代理而不是 1 个通用 Agent？
专业分工，质量可控
每个子代理只做一件事，prompt 精准，行为可预测。code-executor 专注代码执行，不会在分析中途跑偏去写报告。
统计方法怎么保证选对？
确定性规则引擎 + 明确边界标注
正态性检验、方差齐性、检验选择都由 Python 代码按规则执行（100% 可审计，见 ethoinsight/statistics.py）。小样本、边界情况不让 AI "猜"，而是自动打 warning 标签，提示研究员复核。边界定义见上文《能力边界》章节。
领域知识怎么注入？引用如何追溯？
三层证据链 · 每条引用有锚点
方法论规则通过 Skill 静态注入 prompt（skill:// 锚点）；范式参考值来自库内固化阈值表；具体文献通过 MCP 实时检索（noldus-kb://）。每条陈述必须有 source 标注，无法归因的显式标为"探索性推论"。详见《知识检索与证据链设计》。
为什么不是传统软件？
自然语言交互 + 领域推理 + 持续追问
传统软件是固定流水线。Agent 能理解"为什么对照组速度方差这么大？"并结合数据和领域知识给出解释。这是 AI 的本质优势。
