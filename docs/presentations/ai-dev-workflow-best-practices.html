05 — HANDOFF PROTOCOL

Subagent Handoff 协议

Handoff 数据流
code-executor 执行计算
写入 m_*.json（指标）+ groups.json → 调用 seal_handoff 工具 → 生成 handoff_code_executor.json
data-analyst 统计解读
读取指标结果 → 写入统计解读 + 置信区间 + 混淆因素说明 → seal
report-writer 生成报告
读取所有 handoff → 生成 6 章节 APA 报告 → 产出 artifacts → seal
Lead 整合交付
读取所有 handoff → 汇总状态 → 回复用户，附上报告与下载链接
踩过的坑 & 修复方案
Handoff 字段三分裂
metrics / metrics_results / metrics_summary 三种字段名并存，校验器只认一种 → 28/56 条完整 handoff 被误判为残缺
Skill 文档与 System Prompt 冲突
SKILL.md 教 write_file，但 prompt 说"禁止 write_file 必须用 seal tool" → 叙述黑洞，LLM 卡死
Auto-Seal 兜底机制
Subagent 写了文件但忘调 seal → Harness 自动从 artifacts 重建 seal，防止任务卡死
Partial Status 缺失
data-analyst 在 n=1 时应 emit partial，但 Literal 定义缺 partial → Pydantic 拒绝 → 卡死
06 — UPSTREAM SYNC

DeerFlow 上游同步策略

核心原则
默认全量跟随上游
DeerFlow 是底座，不是外部库。默认全合，享受 Bug 修复与性能提升
22 个受保护文件 · Surgical Merge
含 Noldus 定制代码的文件绝不整文件覆盖，必须逐行对比手动合并
关键陷阱：PROTECTED 清单可能滞后
sync 前必须 diff dev-vs-base 所有触及文件，不能只查静态 PROTECTED 清单
受保护文件类型
prompt.py（指令集） agent.py（middleware 链） tools/builtins/__init__.py tools/tools.py（注册表） feedback/sql.py（Schema） sandbox/tools.py（路径翻译）
Prompt 文件合并的致命陷阱
# ❌ 错误：只 grep 关键字确认"没丢定制" grep -n "topOfMind" memory/prompt.py ✓ Found — 误以为定制完好 # 真实案例：grep 串在，但约束被削弱 # BASE 原句（强约束伞句）： "MUST NOT be written into any memory field" # 合并后退化为（窄句）： "Do not record uploads" # 只管 upload，漏掉其他字段 # ✅ 正确做法：三方字节对账 diff <(sed -n '100,200p' base/prompt.py) \ <(sed -n '100,200p' merged/prompt.py) # 看强句计数 / typo / 缩进降格
📌 "grep 串还在"不等于"约束完整"——必须三方字节对比，检查强约束伞句是否被降格为弱约束子句
07 — GUARDRAILS

三层 Guardrail 防护

Layer A · 确定性
指标合法性校验
ethoinsight/validate.py

• NaN/Inf 检测
• 数值范围合理性
• 运行时无 LLM 参与

触发：code-executor 产出后立即校验
Layer B · Catalog-aware
领域约束校验
ethoinsight/validate_catalog.py

• ratio 指标必须在 [0,1]
• count 指标必须 ≥ 0
• 按 output_unit 类型路由校验

依赖：Catalog 元数据驱动
Layer C · 策略授权
工具调用前门控
GuardrailMiddleware

• 未支持范式软拒绝
• EV19 模板确认要求
• Bash 白名单（catalog.resolve）

动作：allow / deny（含指令） / redirect
关键设计原则
Deny 消息必须含明确指令
只写"此操作不允许"不够，LLM 知错不改。必须写"请改调 X 工具，因为 Y，然后做 Z"
Bash 白名单需覆盖所有合法路径
chart-maker 需要 catalog.resolve + dump_headers，但白名单只放行 ethoinsight.scripts.* → 潜伏到 dogfood 才暴露
兜底机制可观测性（反教训）
⚠️ 兜底静默工作太好会把上游回归藏进"管线全绿"

硬性要求：
1. sealed_by 来源标记（区分正常/auto-seal）
2. 触发率写进验收项（skill 修好后应趋近 0）
3. 空心绿比诚实 FAILED 更危险
08 — SINGLE SOURCE OF TRUTH

SSOT 原则 & Catalog 驱动

核心原则
指标定义只存一份
范式映射 / 展示元数据 / 指标定义绝不双存。Skill 文档不得内嵌结构化知识（YAML 格式），知识必须在 ethoinsight 库内
Catalog 驱动 code-executor
Lead 调 catalog.resolve CLI → 生成 metric_plan.json → code-executor 按计划执行 → 无范式知识直接写死在脚本里
Skill 详述工具输出 → LLM 脑补
Skill 写了返回格式 → deepseek 不调工具直接伪造返回值。工具教学只讲"何时调"，绝不讲"返回什么"
不跨范式复用指标
即使两个范式的指标结构相同，也各自独立声明——overlap 接受重复，比跨范式引用引入的歧义风险更安全
parameters_used 字段的教训
# ❌ 错误：把所有输入参数都塞进 parameters_used parameters_used = { "pendulum_duration": 60, # ← 幽灵参数 "velocity_threshold": 5, # ← 幽灵参数 "mobility_state": "mobile" # ← 实际使用的 } # data-analyst 审计 12 turn 发现参数矛盾 → 卡死 # ✅ 正确：只含实际 resolution path 的参数 parameters_used = { "mobility_state": "mobile" # 唯一生效参数 } # 按路径裁剪参数 + signal_key 同源
📌 SSOT 在 review-packages/ 下，而不是 catalog 或 skill——做范式判断前必须先 grep review-packages
09 — DATABASE & DEPLOYMENT

数据库迁移 & 部署细节

数据库迁移策略
Alembic 管理 Schema 变更
SQLAlchemy 模型 → Alembic 自动检测 → 生成迁移脚本 → 启动时应用
deploy 不自动跑迁移！
make deploy-tar 只做 docker load + up，create_all 不给已存在表补列。加列的 sprint 必须手动跑 alembic upgrade
Alembic env.py 特别配置
env.py 不读 -x/env 参数，必须用 set_main_option 直接指定真实 DB 路径 ${DEER_FLOW_HOME}/data/deerflow.db
部署配置对齐原则
dev/prod 行为必须对齐到 docker-compose.yaml
make dev 的 env/CLI flags 是产品属性，不是调试开关。必须复制到 compose，不能只放 .env
多 Worker 切 Thread 的 409 问题
Gateway 多 worker 模式下 StreamBridge 是内存对象，不跨 worker → 切换 thread 时 409。需用进程外 store
CSRF Cookie 生命周期对称
csrf_token 是 session cookie（关标签页即失效），但 access_token 持久 → 重开浏览器后 POST 403。修：加 max_age 对齐 token_expiry_days
10 — AI-ASSISTED DEVELOPMENT

AI 如何辅助项目开发

📋
Spec-First 工作方式
每个 feature 先写 spec 文档（docs/specs/），明确：
• 根因分析
• 修改文件 & 预期行为
• 测试验收标准
• Red anchor test（先红后绿）

Agent 按 spec 实施，不跑题不过度设计
🔁
Handoff 文档传递上下文
跨会话工作通过 handoff 文档传递：
• 已验证的事实
• 阻塞原因
• 已完成 artifacts
• 下一步建议

新 agent 接手前必须先 grep + 解码 checkpoints 核实声称的 trace
🧪
Dogfood 驱动迭代
每个 sprint 完成后跑真实场景 dogfood：
• 验证完整分析流程
• 发现 prompt/schema 矛盾
• 暴露 guardrail 缺口

Golden Cases 作为回归测试基准，同时是 SFT 训练数据种子
关键方法论原则
根因未隔离前别叠加兜底
加新兜底 = 给下次误判铺路。先做决定性实验，隔离根因，再按需加防护
哑故障比响亮故障更危险
空心绿比诚实 FAILED 更糟。兜底的触发率必须可观测，不能静默藏回归
先 curl 实证，再下结论
Cookie / Auth / 网络故障：先 curl -D - 看原始 headers，别靠"看起来像"的 commit 标题猜测
11 — KEY TAKEAWAYS

核心经验总结

工程实践
TDD 是硬性要求，不是建议
先写测试，再实现代码。合并前必跑全量测试 + 裸导入检查
单一事实来源（SSOT）
指标、范式、规则只存一处。Skill 文档只教"何时用"，不内嵌结构化数据
Spec-First，不跑题
写 spec → review → 实施。Agent 只做 spec 指定的事，不加无关"优化"
AI Agent 协作
Middleware 可组合性
每个行为是独立 middleware，顺序即优先级。新功能 = 新 middleware，不修改已有逻辑
响亮失败 > 静默成功
兜底机制必须可观测。sealed_by 来源标记 + 触发率指标是硬性验收项
根因隔离是第一原则
症状相同 ≠ 根因相同。先做决定性实验隔离根因，再按需叠加防护机制
