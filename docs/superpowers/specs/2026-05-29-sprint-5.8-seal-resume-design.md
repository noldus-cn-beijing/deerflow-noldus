## 0. 给实施 agent 的第一性说明（先读，别跳）

### 要解决什么

ethoinsight subagent（code-executor / data-analyst / chart-maker / report-writer）的 LLM 在跑完分析后，有时**没调 `seal_*_handoff` tool 就结束**，导致 handoff 文件不产、下游断链。5.7 已经能**检测**这种情况并标 FAILED（让 lead 重派整个 subagent），但重派整个 subagent 很贵（从头跑分析）。

本 sprint 的核心洞察（经探针实证）：

1. **LLM 在干净/聚焦的上下文里不漏调**——探针证明 deepseek-v4-pro 在简短上下文里会正确"先分析→再调 seal→args 完整"。漏调主要发生在**长链末端**（上下文重、注意力衰减）。
2. **`final_state["messages"]` 包含 subagent 的完整对话历史**——它跑过的分析、结论都在里面，只是没落进 seal tool_call。
3. **不能用 forced tool_choice**（探针证明：deepseek thinking mode 拒绝该参数；关 thinking 后强制调用但 **args 全空**——产空 handoff 比漏调更糟）。
4. **不能用 structured output**（会 strip 掉 seal tool 的 `runtime` 注入参数，撞 `feedback_langchain_tool_args_schema_strips_injected` 教训）。
5. **必须模型无关**——线上现在用 deepseek-v4-pro（dashscope），未来会换 Qwen3（vLLM/Fireworks），thinking/tool_choice 行为各不相同。任何依赖模型特定行为的方案都会在切模型时爆。

### 解决思路（工程化收尾补轮）

在 subagent 的 `astream` 循环**自然结束后、标 COMPLETED 前**，若检测到 handoff 文件未产：
- **不直接标 FAILED**，而是**用 `final_state["messages"]` + 一条聚焦的"现在请调 seal"指令，再 astream 一轮**。
- 这一轮上下文聚焦（"分析已完成，唯一要做的是 seal"），漏调概率极低（探针实证聚焦上下文不漏调），且不强制 tool_choice（args 质量有保障）。
- 补轮后再查一次文件：产了 → COMPLETED；仍没产 → 标 FAILED（**走 5.7 兜底，lead 重派**）。

**这是纵深防御的中间层**：subagent 内部就地补救（便宜，覆盖绝大多数情况）→ 补不上才外层 5.7 兜底（贵，极罕见）。

### 为什么模型无关

"给 LLM 一个聚焦的收尾轮"对任何 LLM 都有效（任何模型在聚焦上下文里都更可能调对工具）。不碰 tool_choice、不碰 structured output、不碰模型 API 层——纯粹是 executor 执行流程的事。换 Qwen 零影响。

### 实施纪律

1. **TDD 强制**（CLAUDE.md 硬规则）：先测后写。
2. **保留 5.7 兜底**：本 sprint 是"在 5.7 标 FAILED 之前插入一次补救"，**绝不删除 5.7 的 FAILED 路径**——补轮失败后仍要走它。
3. **不退化基线**：agent backend 当前 **3133 passed**（含 5.7 的 22 测试），改完必须 ≥3133。
4. **这片区域有教训**（`feedback_langchain_tool_args_schema_strips_injected`）：astream/state/模型调用路径的改动，**异步路径（ainvoke/astream）必须真测**，不能只测同步包装。
5. 只在 dev，commit message 中文。

### ★ 地基已实证（2026-05-29 grill 探针 /tmp/probe_resume.py）

**`create_agent` 第二次 astream 喂"完整历史 messages + 收尾 HumanMessage"的行为已用真实 deepseek-v4-pro 验证 = B（只响应新指令，不重放历史）：**

## 3. 详细实施

### 3.1 ~~增 max_turns 预算~~ — 删除（grill 2026-05-29 查证后否决）

**不改 max_turns。** 原 spec 假设"data-analyst 漏调可能因撞 max_turns=12 被 730 行 break"，并据此 +4。**查证 dogfood thread `439b1405` 的 training-data jsonl 后此假设破产**：

- data-analyst 成功完成的进度时间线是 **`3/4: 调用 seal_data_analyst_handoff`**——只用 3-4 轮 AI message，max_turns=12 余量 8 轮，绰绰有余。漏调那次也在 3-5 轮就漏，远未触及上限。
- 漏调真因是**长上下文末端"以为说了就等于做了"的注意力问题**（输出了"Now let me compile and seal"却没调 tool），与轮数预算无关。
- 补轮是主循环外的独立 astream（见 §3.3），受 recursion_limit 管，**不吃 max_turns 预算**，不需要为它加 buffer。
- 加 max_turns 有真实代价（更多 token/延迟/跑偏空间），无收益的加预算是负优化。

故 **4 个 subagent 的 max_turns 保持原值不动**（code-executor=40 / data-analyst=12 / chart-maker=15 / report-writer=15）。

> 旁注：查证中发现 code-executor 进度时间线满格 8/8、chart-maker 10/10 看似贴上限跑——但它们是**成功**的（没漏调），是否被 max_turns 限制是【独立问题】，不属于 5.8 范围，不在此处理。

### 3.2 executor.py：新增 `_attempt_seal_resume` helper

**补轮次数 = 1（显式决策，grill 锁定 2026-05-29）**

补轮只补 **1 次**，写成显式常量而非隐含写死：

```python
# Sprint 5.8: seal-resume 只补 1 轮。这是深思过的决策，不是任意值。
# 理由：补轮失败的主因更可能是【系统性】(分析结论本身残缺 / LLM 卡在理解错误)
# 而非【随机】——再补一轮多半是同样结果。补不上时，交给 5.7 兜底重派【整个
# subagent】(全新上下文)反而更可能成功。纵深防御已足够：补轮(便宜捞一次) →
# 5.7 重派(全新上下文) → lead 最多 2 次 retry。不需要在补轮内部再加循环。
# 改大此值的代价：消耗更多 turn + token + 延迟（补轮是独立 astream，受 recursion_limit 管）。
_SEAL_RESUME_MAX_ATTEMPTS = 1
```

**位置**：`SubagentExecutor` class 内（紧邻其他方法，如 _build_initial_state 之后）。

设计要点：
- 接收 `agent`（已创建的 graph）、`final_state`（自然结束时的 state）、`run_config`、`context`、`result`、`collector`。
- 构造 `resume_messages = final_state["messages"] + [HumanMessage(聚焦收尾指令)]`。
- 聚焦指令用**正面措辞**（CLAUDE.md §6 deepseek 正面提示原则）：明确"分析已完成，现在调用 seal_X_handoff 把结论落库"，**不要**写"不要忘记/禁止只在 thinking 写"。
- 再 astream 一轮，收集新 ai_messages（复用现有去重逻辑），更新 final_state。
- **绝不抛异常**（同 5.7 R1 原则）——补轮本身失败（astream 报错）不能让整个 task 崩，捕获后让外层走 5.7 FAILED。
- 返回更新后的 final_state（或 None 表示补轮异常）。

> **⚠️ 实施提醒（training-data 影响，grill 2026-05-29 标记，不强制在 5.8 解决）**：补轮注入的收尾 `HumanMessage` 是 **harness 内部机制**，不是真实用户/lead 输入。`TrainingDataMiddleware`（录 jsonl 供 SFT）若把它当正常对话录入，会污染训练数据——让模型学到"分析中途会有人催 seal"这种现实中不存在的模式（SFT 是 v0.1 核心路线，数据质量要紧）。实施时**确认补轮的注入消息不进 training-data**（或确认 TrainingDataMiddleware 的过滤逻辑天然排除它）。若当前实现已有重复录制问题（5-29 查 jsonl 见 code-executor/data-analyst 被重复录 3 次），这是相邻的已知问题，可一并记录但不必在 5.8 修。

```python
async def _attempt_seal_resume(
    self,
    agent,
    final_state: dict,
    run_config,
    context: dict,
    result,
    collector,
) -> dict | None:
    """漏调 seal 时，就地补一轮聚焦收尾。返回更新后的 final_state，异常时返回 None。

    不强制 tool_choice（探针证明会产空 args），不用 structured output（会 strip
    runtime 注入）。只是给 subagent 一个上下文聚焦的"现在请 seal"轮次——探针实证
    聚焦上下文里 LLM 会正确调 seal 且 args 完整。模型无关（Qwen 切换零影响）。

    ROBUSTNESS: 绝不抛异常。补轮失败 → 返回 None → 外层走 5.7 FAILED 兜底。
    """
    seal_tool = f"seal_{self.config.name.replace('-', '_')}_handoff"
    # 补轮 prompt 措辞（grill 2026-05-29 锁定）：用 HumanMessage 追加（不是第二个
    # SystemMessage——dashscope/create_agent 不接受中途插第二条 SystemMessage，见
    # executor.py:567 已踩坑规避）。措辞保持下方现状即可：它已是"指令 + 终结步骤"
    # 框定（"这是最后一步"），且 2026-05-29 E2E dogfood 实证补轮用它成功调对 seal。
    # 【实施 agent 不要"优化"这段措辞】——能跑通就不动；正面措辞，勿加"不要重新分析/
    # 不要解释"之类反向激活词（CLAUDE.md §6 deepseek 正面提示原则）。
    resume_prompt = (
        f"你上面的分析已经完成。现在请调用 {seal_tool} 工具，"
        f"把你刚才得出的结论（key_findings / 各结构化字段）填入工具参数并落库。"
        f"这是完成本次任务的最后一步——调用该工具后任务即完成。"
    )
    try:
        from langchain_core.messages import HumanMessage  # 顶部已 import，此处示意
        messages = list(final_state.get("messages", []))

        # Sprint 5.8 前置守卫（grill 锁定 2026-05-29）：必须有实质分析内容
        # (至少一条 AIMessage) 才值得补轮。补轮的全部价值是"让 LLM 把【已经做好的】
        # 分析 seal 掉"；若 history 里连一条 AIMessage 都没有(subagent 根本没产生过
        # 实质分析，如第一轮就被截断/异常早退)，补轮无米下锅——强行补只会让 LLM 用空
        # args 调 seal，产出空 handoff(比漏调更糟)。这种情况返回 None，交给 5.7
        # 兜底重派【整个 subagent】(全新跑一次)。语义：只补"漏了 seal"，不补"没分析"。
        if not any(isinstance(m, AIMessage) for m in messages):
            logger.warning(
                "[trace=%s] Subagent %s: no AIMessage in history, skip seal-resume "
                "(no analysis to seal; let 5.7 re-dispatch)",
                self.trace_id, self.config.name,
            )
            return None

        messages.append(HumanMessage(content=resume_prompt))
        resume_state = {**final_state, "messages": messages}
        # Sprint 5.8 显式重注入 thread_data + sandbox（grill 2026-05-29 锁定）：
        # 补轮里 subagent 调 seal_*_handoff 时，seal tool 从 runtime.state["thread_data"]
        # ["workspace_path"] 取目录写文件（seal_handoff_tools.py:_resolve_workspace）。
        # 探针(/tmp/probe_thread_data.py)已实证 graph 的 final_state 确实透传了
        # thread_data，所以【当前】不注入也能 seal。但仍显式重注入——零成本(executor 手里
        # 就有 self.thread_data/self.sandbox_state)、不依赖"graph 透传自定义 state 字段"
        # 这个 langchain 版本相关的隐式行为(版本/middleware 变动可能改变)。显式 > 隐式透传
        # ([[feedback_dev_prod_behavior_alignment]] 精神)。缺 workspace 时 seal 会抛
        # RuntimeError → 被 try/except 吞 → 补轮静默白做，极难诊断，故防御性补这一行。
        if self.thread_data is not None:
            resume_state["thread_data"] = self.thread_data
        if self.sandbox_state is not None:
            resume_state["sandbox"] = self.sandbox_state

        logger.info(
            "[trace=%s] Subagent %s: handoff missing, attempting seal-resume (1 focused turn)",
            self.trace_id, self.config.name,
        )
        new_final = None
        async for chunk in agent.astream(
            resume_state, config=run_config, context=context, stream_mode="values"
        ):
            if result.cancel_event.is_set():
                return new_final  # 尊重取消
            new_final = chunk
            # 收集补轮的 AI message（复用主循环的去重模式）
            msgs = chunk.get("messages", [])
            if msgs and isinstance(msgs[-1], AIMessage):
                md = msgs[-1].model_dump()
                mid = md.get("id")
                dup = (any(m.get("id") == mid for m in result.ai_messages)
                       if mid else md in result.ai_messages)
                if not dup:
                    result.ai_messages.append(md)
        return new_final if new_final is not None else final_state
    except Exception:
        logger.exception(
            "[trace=%s] Subagent %s: seal-resume attempt failed; falling back to 5.7 FAILED",
            self.trace_id, self.config.name,
        )
        return None
```

> **关键**：补轮的 astream **不受 730 行 max_turns break 约束**——因为它是主循环外的独立一次 astream，主循环已经退出。补轮内部若 subagent 又跑多轮（理论上不会，因为指令聚焦），靠 recursion_limit 兜底。max_turns 本身**不改**（§3.1 查证后否决：分析只用 3-4 轮，远未触及上限）。

                self.thread_data.get("workspace_path")
                if isinstance(self.thread_data, dict)
                else None
            )
            _handoff_error = _validate_handoff_emitted(self.config.name, _workspace)

            # Sprint 5.8: 漏调 seal → 先就地补一轮聚焦收尾，补不上才走 5.7 FAILED。

                # 补轮后重新校验。

                # 产垃圾。探针实证补轮 args 完整(概率低)，且"空内容 handoff"是 seal tool
                # 的【普遍】问题(正常调用也可能偶发，非补轮引入)，在补轮里单独堵不治本。
                # 5.8 不处理；留给独立的「handoff 内容非空/schema 完整性校验」后续 fix

            if _handoff_error is not None:
                logger.warning(

                    "(seal-resume did not recover): %s",
                    self.trace_id, self.config.name, _handoff_error,

                result.try_set_terminal(SubagentStatus.COMPLETED)

**关键点**：
- 补轮只在 `_handoff_error is not None`（白名单 + 文件缺失）时触发——非白名单 subagent（general-purpose/bash）`_validate` 返回 None，不会补轮。
- `agent` / `run_config` / `context` / `collector` 这几个变量在 `_aexecute` 作用域内可见（694 行 astream 用的就是它们）——**已核验**：`agent`=644行 / `collector`=651行 / `run_config`=667行 / `context`=672行，全在 `_aexecute`(618-836) 内且赋值早于 807 行检查点，补轮可直接复用。
- 补轮失败（返回 None）→ final_state 不变 → 重新校验仍失败 → 走 FAILED（5.7 兜底）。
- **与 5.7 lead retry 不打架（grill 2026-05-29 确认）**：5.7 的 lead retry 触发是 AND 条件——`task failed` **且** error 含 `"terminated without emitting"`（prompt.py:254）。补轮**成功** → COMPLETED + 无 error → 两条件都不满足 → lead 不重派（无双重补救）。补轮**失败** → FAILED + 该 error → 满足 → lead 重派（兜底链）。两机制按 COMPLETED/FAILED 天然隔离，无需额外协调。

### 3.4 token_usage / collector 处理

补轮会产生额外 token 消耗。`result.token_usage_records = collector.snapshot_records()` 在 735 行（主循环后）已执行一次。补轮后需要**重新 snapshot**，确保补轮的 token 也计入。实施时在补轮逻辑后补一行 `if collector is not None: result.token_usage_records = collector.snapshot_records()`。

### ⚠️ 已知未覆盖失败模式（grill 2026-05-29 锁定 B，必须后续修复）

**"空内容 handoff"**：当前 `_validate_handoff_emitted` 只查文件存在，不查内容。LLM（正常调 seal 或补轮被催着调）极小概率会调了 seal 但核心字段残缺/空（如 `key_findings=[]`、`summary=""`）→ 文件产出 → 判 COMPLETED → 下游读到空内容 handoff，无人报警、一路绿灯产垃圾。

- **为何 5.8 接受现状**：(1) 探针实证补轮 args 完整，概率低；(2) 这是 seal tool 的【普遍】问题（forced tool_choice 会必然产生、正常调用也可能偶发），非补轮引入的新问题，在补轮路径单独堵不治本；(3) 为 4 个 subagent 各定义"核心字段非空"判据是另一个 sprint 量级，塞进 5.8 是 scope creep。
- **后续 fix（已立案，必须做）**：新开「handoff 内容完整性校验」sprint —— 增强 `_validate_handoff_emitted`（或新建 validator）按各 subagent 契约检查核心字段非空，覆盖正常路径 + 补轮路径。届时 5.8 的补轮判定会自动受益（因为复用同一 validator）。

## 9. 模型无关性说明（为未来 Qwen 切换背书）

本方案**不依赖任何模型特定行为**：
- 不用 `tool_choice`（deepseek thinking 拒绝、各后端行为不同）
- 不用 `with_structured_output`（strip 注入参数）
- 不读模型特定的 reasoning 字段
- 只做"给 subagent 一个聚焦的收尾轮"——这对 deepseek-v4-pro（dashscope）、Qwen3（vLLM/Fireworks）或任何 LLM 都成立

线上从 deepseek 切到 Qwen 时，本机制**无需改动**。这是相对 forced tool_choice 方案的决定性优势（后者每换一次模型/后端都要重验重写）。
