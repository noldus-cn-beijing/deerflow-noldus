## 1. 背景与目标

### 问题来源（2026-05-28 dogfood 实证）

EPM n=1 thread `68f6da40-6c62-494e-922c-8e3c82c156b5` row 14：data-analyst 第二次派遣（用户 acknowledge_quality=True 后）的 task 状态 **COMPLETED 但 `handoff_data_analyst.json` 没产**。

具体过程：
```
轮 1-3: read_file (constitution / handoff_code_executor / paradigm md)  ← tool_call
轮 4: "现在我已掌握所有必要信息,来进行综合分析..."（千余字 thinking）
      "现在封存分析结果:"                                                 ← 只有 content,无 tool_call
轮 5: 无
```

executor 见 AI message 无新 tool_call → 标 task COMPLETED → 收工。但 LLM **完全没调用** `seal_data_analyst_handoff` tool，`/mnt/user-data/workspace/handoff_data_analyst.json` 不存在。

下游 chart-maker / report-writer 通过 `read_file handoff_data_analyst.json` 拿数据，文件不存在则整条链断。当前 lead 是通过事后 `read_file` 探测 + thinking 推理 "似乎没有被正确封存" 后**自救式重新派遣**——但这条 fallback 路径依赖 lead 的运气，且用户能看到内部错误诊断。

### 根因（已确认 + 已部分修复）

Sprint 1 实施时在 `data_analyst.py:60-94` workflow 插入 step 2.5/2.6，但原 step 2 "一次性完成核心分析推理" 没改成 step 2.7，导致两个 "step 2" 共存。LLM 把第二个 step 2 理解为 "在 thinking 里输出全部结论就算完成"，跳过了 step 3 的 seal tool_call。

**已修复（commit `2df10880`）**：第二个 step 2 → step 2.7。**但这是 prompt 层 fix，不能根治** —— LLM 随机性意味着将来其他 prompt 改动、其他 subagent、其他长链 reasoning 都可能复现"漏调 tool 静默成功"的同类 bug。

### Sprint 5.7 目标

**Harness 层硬约束**：让 executor 在 set_terminal(COMPLETED) 之前**强制**校验 ethoinsight 四个核心 subagent (`code-executor` / `data-analyst` / `chart-maker` / `report-writer`) 的对应 handoff 文件是否落库。文件不存在 → 标 task FAILED + 明确 error message → lead 收到失败信号触发自动重试。

### 设计原则

1. **harness 强约束 > prompt 配方**（[[feedback_deny_messages_must_direct]] / [[project_2026-05-18_lead_not_reading_handoff]] 教训：prompt-only fix 对关键路径不够）
2. **正交于 Sprint 5.5 lineage 封印**：本 sprint 管"handoff 存在性"，Sprint 5.5 管"handoff 内容一致性（sha256）"。前者是后者的前提
3. **white-list 而非 black-list**：只对明确产 handoff 的 4 个 ethoinsight subagent 强制；general-purpose / bash / knowledge-assistant 不验证
4. **fail loudly with diagnostic**：error message 必须含具体 subagent 名 + 期望文件名 + "LLM forgot to call seal_*_handoff tool" 诊断，便于 lead 识别 retry 类别
5. **idempotent**：检查仅在 set_terminal(COMPLETED) 前触发；CANCELLED / FAILED / TIMED_OUT 路径不动

## 2. 文件改动清单

### 2.1 新增 `_validate_handoff_emitted` helper

**位置**：`packages/agent/backend/packages/harness/deerflow/subagents/executor.py`（顶部 helpers 区，约 line 100 附近）

```python
# Ethoinsight subagents that MUST seal a handoff file before terminating successfully.
# Keys are subagent config.name; values are expected handoff filename in workspace.
# General-purpose subagents (bash / knowledge-assistant / general-purpose) are NOT
# in this map — they don't have a seal_*_handoff contract.
_HANDOFF_EMISSION_REQUIRED: dict[str, str] = {
    "code-executor": "handoff_code_executor.json",
    "data-analyst": "handoff_data_analyst.json",
    "chart-maker": "handoff_chart_maker.json",
    "report-writer": "handoff_report_writer.json",
}


def _validate_handoff_emitted(
    subagent_name: str,
    workspace_path: str | None,
) -> str | None:
    """Return None if the subagent's handoff file is present, else a diagnostic.

    Called just before setting SubagentStatus.COMPLETED. If the subagent is in
    _HANDOFF_EMISSION_REQUIRED but its handoff file is absent, the returned
    string explains the failure so set_terminal(FAILED, error=...) can use it.

    Args:
        subagent_name: SubagentConfig.name (e.g. "data-analyst").
        workspace_path: Host-side workspace path from ThreadDataState. None when
            ThreadDataMiddleware didn't set it (old threads / dev paths).

    Returns:
        None if validation passes or doesn't apply; otherwise an error string.
    """
    expected_filename = _HANDOFF_EMISSION_REQUIRED.get(subagent_name)
    if expected_filename is None:
        # Subagent not in white-list — no contract to enforce.
        return None

    if not workspace_path:
        # No workspace resolvable — can't validate, fail-open (do not block).
        logger.warning(
            "Subagent %s: cannot validate handoff emission (no workspace_path); "
            "skipping validation.",
            subagent_name,
        )
        return None

    handoff_path = Path(workspace_path) / expected_filename
    if handoff_path.exists():
        return None

    return (
        f"Subagent '{subagent_name}' terminated without emitting "
        f"'{expected_filename}'. This means the LLM finished its reasoning "
        f"but forgot to call seal_{subagent_name.replace('-', '_')}_handoff "
        f"tool. The handoff file is REQUIRED by downstream subagents. "
        f"Lead should re-dispatch this task with an explicit reminder to "
        f"invoke the seal tool at the end."
    )
```

### 2.2 改 `try_set_terminal(COMPLETED)` 调用点

**位置**：`packages/agent/backend/packages/harness/deerflow/subagents/executor.py:723`

**改前**：
```python
result.try_set_terminal(SubagentStatus.COMPLETED)
```

**改后**：
```python
# Sprint 5.7: validate handoff emission before marking COMPLETED.
# For ethoinsight subagents (code-executor / data-analyst / chart-maker /
# report-writer), the expected handoff file MUST exist in workspace. LLM
# can finish its thinking turn without emitting the seal tool_call —
# executor would otherwise mark task COMPLETED falsely.
handoff_error = _validate_handoff_emitted(
    subagent_name=self.config.name,
    workspace_path=self._resolve_workspace_path(),
)
if handoff_error is not None:
    logger.warning(
        "[trace=%s] Subagent %s terminated without emitting handoff: %s",
        self.trace_id, self.config.name, handoff_error,
    )
    result.try_set_terminal(SubagentStatus.FAILED, error=handoff_error)
else:
    result.try_set_terminal(SubagentStatus.COMPLETED)
```

### 2.3 新增 `_resolve_workspace_path` helper

**位置**：`SubagentExecutor` class 内（紧跟现有 helpers）

```python
def _resolve_workspace_path(self) -> str | None:
    """Resolve host-side workspace_path from executor's thread_data.

    Mirrors deerflow.agents.middlewares.experiment_context.resolve_workspace_from_state
    but works from executor's stored thread_data dict (which the lead-agent
    middleware chain set up).
    """
    thread_data = self.thread_data
    if not isinstance(thread_data, dict):
        return None
    workspace = thread_data.get("workspace_path")
    return workspace if isinstance(workspace, str) and workspace else None
```

**注**：`self.thread_data` 已在 `SubagentExecutor.__init__` 接收（task_tool.py:498 `"thread_data": thread_data` 传入）。本 helper 只是把 dict 取值 + 类型守护。

## 4. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 老 thread（无 workspace_path）跑到验证逻辑 → fail-open 不阻断；但若同时 LLM 真漏 seal，会**静默成功**与 sprint 5.7 前一致 | T4 case `test_no_workspace_path_skipped` 覆盖 + 日志 warn 留痕；不引入 regression |
| code-executor 在 sandbox 内 seal 后，executor 验证用的是 host workspace_path —— 路径不一致 | seal_*_handoff tool 已经走 ThreadState.workspace_path（Sprint 0 spec），与 executor 用同一来源；T1 case 验证 |
| 多 subagent 并发（最多 3）时同一 workspace 可能被 race | 单个 thread workspace 是同 thread 各 subagent 顺序使用，task tool 并发也是不同 task_id；handoff 文件名按 subagent 区分，不互相覆盖；不引入新 race |
| seal_*_handoff 调用成功但 atomic write 失败（_seal_handoff 内部抛异常）→ task 提前 FAILED 已有 path | Sprint 0 _seal_handoff 已有 raise，本 sprint 不重叠 |
| lead retry 逻辑陷入循环（每次都触发漏 seal）| §2.4 prompt 限制最多 2 次重试 + lead 报错给用户；2 次仍失败几乎可确认是 prompt 配置问题，需人工介入 |
| general-purpose subagent 临时被用来做 ethoinsight 工作（"我让 general-purpose 算指标"）| 这种用法本就是反模式（违反 lead 派遣边界）；本 sprint 不强行扩验证，避免误伤合法 general-purpose 用例 |
| ChartMakerHandoff 文件名是否 "handoff_chart_maker.json" 还是其他 | T1 实施前 grep `handoff_registry.py` 确认实际文件名；当前已知是 `handoff_chart_maker.json`（HANDOFF_FILE_REGISTRY 第 16 行）|

## 7. 与 Sprint 5.5 的协同

| | Sprint 5.5 | Sprint 5.7 |
|---|---|---|
| **管什么** | handoff 内容一致性（sha256 hash 验真）| handoff 文件是否存在（emission 验真）|
| **触发点** | 下游 subagent 派遣**之前**（pre-tool-call guardrail）| 当前 subagent 终结**之前**（executor 内部）|
| **机制** | `LineageIntegrityGuardrailProvider.evaluate()` 拦 task | `_validate_handoff_emitted` 在 set_terminal 前调 |
| **失败处置** | guardrail 返回 deny ToolMessage → lead 收到拒绝 | executor 标 task FAILED → lead 收到失败 |
| **本质** | 防"上游 handoff 被污染" | 防"上游 handoff 根本没生成" |

**正交且互补**：Sprint 5.7 是 Sprint 5.5 的前提（先确保文件存在，再校验内容完整）。两者可以独立实施、独立合并。

如果 Sprint 5.7 先合，Sprint 5.5 启动时可以**假定** handoff 文件存在；如果 Sprint 5.5 先合，它的 guardrail 第一行先检查文件存在性（实际上 Sprint 5.5 spec 也已经隐含这个 precondition）。
