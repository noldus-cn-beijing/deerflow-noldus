## 0. 给实施 agent 的第一性说明

### 要解决什么

5.7 的 `_validate_handoff_emitted` + 5.8 的补轮判定**只查 handoff 文件是否存在，不查内容**。LLM（正常调 seal、或 5.8 补轮被催着调）极小概率会**调了 seal 但核心字段残缺/空**（`key_findings=[]`、`summary=""`、`metrics_summary={}`）→ 文件产出 → 判 COMPLETED → 下游 chart-maker/report-writer 读到空内容 handoff → **无人报警、一路绿灯产垃圾**。这比漏调更隐蔽：漏调至少 5.7 会响亮 FAILED，空内容 handoff 静默通过。这是真实会发生的失败模式（seal tool 的普遍问题，非补轮引入）。

### 为什么是"内容非空"而不是"sha256 hash 封印"

本 sprint 原设计是 handoff 内容 sha256 封印 + fail-closed + audit log（防篡改）。**2026-05-29 grill 复审否决**（见 roadmap「grill 复审」表 5.5 行 + ADR-0001）：
- 威胁不存在——单产品、确定性脚本、自有 subagent 链，没有"恶意篡改 handoff"的威胁模型
- "写一半被读"的 race 已被 Sprint 0 的 atomic write（tmp + os.rename，POSIX 原子）覆盖
- 用户/demo 无感（公测标准是"6 范式跑得准"，hash 封印不贡献）
- **真需求是"内容非空"**（空 handoff 静默产垃圾，实际发生）；hash 防的是假需求

所以本 sprint 只做"内容非空校验"，hash 封印推 v0.2 或砍。

### 设计原则

1. **复用 5.7 的 validator 路径**：在 `_validate_handoff_emitted` 的"文件存在"检查（executor.py:146 `if handoff_path.exists()`）**之后**追加"核心字段非空"检查。文件存在但核心字段空 → 返回诊断（而非 None）→ 走 5.7 FAILED / 5.8 补轮。**一处改动同时覆盖正常路径 + 补轮路径**（5.8 补轮复用同一 validator，自动受益）。
2. **核心字段判据来自 handoff 契约（single-source）**：各 subagent 的"核心字段"从 `handoff_schemas.py` 的 Pydantic 类语义推导，**不在 executor 里硬编码一份字段名清单**（否则与 schema 双存，违反 [[feedback_single_source_of_truth]]）。实现上用一个 `{subagent_name: 核心字段检查函数}` 映射，紧挨 `_HANDOFF_EMISSION_REQUIRED`，并注释"判据对应 handoff_schemas.py 的哪个字段"。
3. **绝不抛异常**（继承 5.7 R1）：读 JSON / 解析失败 → fail-open（当作通过，return None），不能让校验本身崩掉正常 task。读坏 JSON 视为"无法判定内容"→ 放行（保守，不阻断）。
4. **不做 hash、不做 audit log、不做 guardrail**（那是被否决的 v0.2 范畴）。失败处理与 5.7 完全一致：返回诊断字符串，executor 标 FAILED。

### 实施纪律

1. TDD 先写测试再写实现。
2. **不退化基线**：dev 当前 **3145 passed**（主仓环境），改完 ≥3145。
3. 校验函数异常安全（同 5.7）。
4. 只在 dev，commit message 中文。

## 3. 详细实施

### 3.1 加核心字段检查映射

**位置**：executor.py，紧挨 `_HANDOFF_EMISSION_REQUIRED`（约 86 行后）。

```python
# Sprint 5.5: 各 subagent handoff 的「核心字段非空」判据。
# 判据语义对应 handoff_schemas.py 的字段（single-source：字段含义以 schema 为准，
# 此处只编码"哪个字段必须非空"，不复制 schema 定义）：
#   code-executor : metrics_summary 非空 dict（CodeExecutorHandoff.metrics_summary）
#   data-analyst  : key_findings 非空 list（DataAnalystHandoff.key_findings）
#   chart-maker   : chart_files 非空 list 或 failed_charts 有说明（ChartMakerHandoff）
#   report-writer : report_path 非空 str（ReportWriterHandoff.report_path）
# 每个检查函数收 parsed handoff dict，返回 None=通过 / str=缺失字段说明。
def _check_code_executor_content(d: dict) -> str | None:
    if not d.get("metrics_summary"):
        return "metrics_summary is empty"
    return None

def _check_data_analyst_content(d: dict) -> str | None:
    if not d.get("key_findings"):
        return "key_findings is empty"
    return None

def _check_chart_maker_content(d: dict) -> str | None:
    # chart_files 空但 failed_charts 有说明 = 合法（图表确实没生成但记录了原因）
    if not d.get("chart_files") and not d.get("failed_charts"):
        return "both chart_files and failed_charts are empty"
    return None

def _check_report_writer_content(d: dict) -> str | None:
    if not d.get("report_path"):
        return "report_path is empty"
    return None

_HANDOFF_CONTENT_CHECKS: dict[str, "Callable[[dict], str | None]"] = {
    "code-executor": _check_code_executor_content,
    "data-analyst": _check_data_analyst_content,
    "chart-maker": _check_chart_maker_content,
    "report-writer": _check_report_writer_content,
}
```

### 3.2 在 `_validate_handoff_emitted` 插入内容检查

**位置**：executor.py:146，`if handoff_path.exists(): return None` 这一处。

**改前**：
```python
    try:
        handoff_path = Path(workspace_path) / expected_filename
        if handoff_path.exists():
            return None
    except Exception:
        logger.exception(...)
        return None
```

**改后**：
```python
    try:
        handoff_path = Path(workspace_path) / expected_filename
        if handoff_path.exists():
            # Sprint 5.5: 文件存在 → 再查核心字段非空。空内容 handoff（调了 seal
            # 但字段残缺）会静默通过下游产垃圾，比漏调更隐蔽。判据见 _HANDOFF_CONTENT_CHECKS。
            content_check = _HANDOFF_CONTENT_CHECKS.get(subagent_name)
            if content_check is None:
                return None  # 无内容判据（理论上白名单内都有）→ 仅凭存在性放行
            try:
                import json
                parsed = json.loads(handoff_path.read_text(encoding="utf-8"))
            except Exception:
                # 读/解析失败 → 无法判定内容 → fail-open 放行（保守，不阻断正常 task）
                logger.warning(
                    "[handoff_content] Subagent %s: cannot parse %s for content check; "
                    "passing on existence only.",
                    subagent_name, expected_filename,
                )
                return None
            missing = content_check(parsed) if isinstance(parsed, dict) else "handoff is not a JSON object"
            if missing is None:
                return None  # 文件存在 + 核心字段非空 → 通过
            # 文件存在但核心字段空 → 落到下面的诊断（fall through）
            seal_tool = f"seal_{subagent_name.replace('-', '_')}_handoff"
            return (
                f"Subagent '{subagent_name}' sealed '{expected_filename}' but its "
                f"core content is incomplete: {missing}. The handoff exists but is "
                f"unusable by downstream subagents. Lead should re-dispatch this task "
                f"reminding it to fill the {seal_tool} arguments with actual results."
            )
    except Exception:
        logger.exception(...)
        return None
```

**关键点**：
- 内容检查整体包在原有 try/except 内（异常安全继承）。
- 读 JSON 失败单独 fail-open（不阻断）——读坏文件不该让正常 task 崩。
- 诊断字符串含 `sealed ... but ... incomplete` + `core content`——**与 5.7 的 "terminated without emitting" 区分**（不同失败类型，便于 lead/日志辨别）。
- 注意：5.7 lead retry 规则（prompt.py）只匹配 `"terminated without emitting"`。**本 sprint 的新诊断 lead 不会自动 retry**——这是设计选择：空内容 handoff 大概率是 LLM 理解/数据问题（重派未必好），先标 FAILED 让 lead 按"任何 subagent 失败必须 ask_clarification"（prompt.py 规则 5）处理。**若要 lead 自动 retry 空内容，需在 prompt 加关键字识别——本 sprint 不做，标注为可选增强。**

## 8. 与 5.7/5.8 的关系

```
subagent 结束 → handoff 文件在？
  ├─ 不在 → 5.7 "terminated without emitting" → 5.8 补轮 → 仍不在 → FAILED
  └─ 在 → 【Sprint 5.5】核心字段非空？
       ├─ 非空 → COMPLETED ✅
       └─ 空 → "sealed but incomplete" → FAILED（lead ask_clarification）
```

5.7 管"有没有产"、5.5 管"产的有没有内容"、5.8 管"漏产时就地补救"。三者复用同一个 `_validate_handoff_emitted` 返回值契约（None=过 / str=失败），叠加无缝。
