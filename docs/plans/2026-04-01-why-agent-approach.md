# EthoInsight：为什么我们选择 Agent 方案

## 我们要解决的问题

Noldus 卖的是行为追踪系统。EthoVision 记录动物去了哪里、做了什么。但追踪只是工作的一半——另一半是理解数据意味着什么。

今天，这种理解完全依赖研究者本人。一个有二十年经验的神经科学 PI 看到高架十字迷宫数据，立刻就知道：实验组的开放臂时间显著低于对照组，结合运动量正常，说明处理增加了焦虑样行为。一个刚入行的实验技术员看到同样的数据，看到的是一张满屏数字的表格。

数据采集和数据解读之间的鸿沟，是大多数实验人员卡住的地方。不是因为统计方法难——t 检验就是 t 检验——而是因为知道**该看哪些指标做组间比较**、**该用什么统计检验**、**效应量说明了什么**、**怎么排除运动量等混杂因素**、**怎么把多个范式的结果串起来**，需要多年的领域经验，而大多数实验人员没有这些经验。

## 原始 PRD 提出了什么

原始产品方案（2026 年 1 月）是一个交互式 Web 可视化平台：轨迹热图、象限分布环形图、学习曲线动画、焦虑指数仪表盘、跨范式雷达图。十六种行为范式。Three.js 做 3D 渲染。Deck.gl 做大规模轨迹可视化。Seaborn 做出版级图表。四级预警的实时监控系统。响应式移动端布局。

方案很宏大，很完整，但瞄准了错误的用户。

## 原始方案的问题在哪

PRD 描述了四个用户画像：神经科学研究员、药理学家、行为学课题组负责人、行为判别技术员。四个用户故事都隐含同一个假设——用户已经知道自己在找什么，只是需要一个更好的方式去看到它。"我希望立即看到轨迹热图，以便快速判断空间策略。""我希望系统自动生成焦虑指数仪表盘，以便比较不同药物组。"

这个假设制造了一个矛盾。PRD 说目标是"将复杂、多维度的动物行为原始数据，转化为直观、可操作的深度洞察"。但交互式仪表盘不转化任何东西。它只是展示。真正的转化——决定哪些指标重要、该用什么统计检验、阈值怎么设、怎么给表型分类——仍然发生在研究者的脑子里。

对于经验丰富的 PI，一个好的仪表盘确实有用。对于第三次跑高架十字迷宫实验的技术员，它只是又一个不知道怎么用的工具。而技术员的数量远多于 PI。

还有现实的问题：

**市场上已经有仪表盘了。** GraphPad Prism 做组间比较。R 和 Python 做定制分析。EthoVision 自己也导出基本统计。再建一个可视化平台，意味着在功能上和已经发展了几十年的工具竞争。

**工程量巨大，回报不确定。** 十六种范式乘以多种图表类型乘以交互功能乘以响应式布局。开发十二个月，第一阶段还只能覆盖三个范式。最难的部分——把 Noldus 的领域专业知识编码进系统——根本没有被解决。PRD 描述了**可视化什么**，但没有描述**怎么解读**。

**交互探索假设用户是专家。** PRD 把"交互探索"列为核心设计原则。但下钻控件、可配置图表类型、统计检验选择器，给最不需要它们的用户增加了复杂度。知道数据不符合正态分布所以该选 Kruskal-Wallis 检验的研究者，不需要系统帮忙。不知道的那个，会选错检验，或者干脆跳过统计分析。

## 我们改做什么

一个端到端的分析 Agent。用户上传 EthoVision 原始数据，选择跑的是哪个范式，拿到一份完整的分析报告——指标、统计、表型分类、自然语言解读。不需要配置图表。不需要选择统计检验。不需要查阈值。

Agent 做的事和一个有经验的研究者一样：看数据，跑对的检验，把结果和已有的参考范围比较，检查混杂因素，写出结论。它不是靠 LLM 即兴发挥，而是沿着一条确定性的流水线执行，流水线建立在结构化的领域知识之上。

关键的架构决策：**LLM 不做科学计算。** 所有指标计算用 scipy。所有统计检验选择遵循 YAML 中编码的规则。所有组间比较使用经过验证的统计方法。LLM 唯一的工作是把结构化的分析结果写成人能读的报告——以及回答用户对结果的追问。

这个分离很重要。科学分析必须可复现。同样的数据必须永远产出同样的指标、同样的 p 值、同样的效应量。LLM 做不到这一点。确定性代码可以。但确定性代码没法用通俗的语言向技术员解释，为什么实验组相比对照组表现出显著更低的开放臂探索时间（p<0.01, Cohen's d=1.2），以及这在生物学上意味着什么。LLM 可以。

## 为什么这个方案适合我们的用户

Noldus 系统的真实用户不是购买系统的 PI。是每天跑实验的研究生、博后和技术员。这些人了解他们的动物和实验方案。他们不一定了解数据分析。

一个技术员跑完高架十字迷宫实验，不想交互式地探索数据。他们想知道：各组之间有没有显著差异？哪些指标的组间差异最大？效应量够不够强？需不需要向 PI 汇报什么？

Agent 直接回答这些问题。仪表盘让用户自己去找答案。

这也恰好发挥了 Noldus 真正的竞争优势。Noldus 有几十年的行为科学专业积累——每种范式哪些指标重要、不同品系的正常范围是多少、怎么检测混杂因素、怎么解读跨范式结果。在原始 PRD 中，这些知识存在文档里，用户可能读也可能不读。在 Agent 方案中，它被编码为结构化知识，系统在每次分析中主动应用。

没有竞争对手能轻易复制这一点。谁都能做一个图表库。不是谁都能做一个知识库，准确地知道每种范式该比较哪些指标、该用什么统计方法、怎么检查运动量等混杂因素、怎么评估效应量的生物学意义，并引用正确的文献。

**之前（错误）：** 有绝对阈值 → 系统按阈值判定"正常/轻度焦虑/重度焦虑"
**实际：** 没有常模 → 核心是对照组 vs 实验组（不同剂量/处理）的组间相对比较

这影响的是知识库的设计。知识库存储的不是"某个指标低于多少就是异常"，而是：每种范式该计算哪些指标、每个指标的生物学含义、组间比较时该选什么统计方法、需要检查的混杂因素、效应量的解读参考。

Agent 的工作也相应调整：不是"判定这只动物焦虑不焦虑"，而是"告诉实验人员各组之间哪些指标存在显著差异、差异有多大、可能的生物学含义是什么"。

## 我们放弃了什么

透明度。交互式仪表盘让用户看到一切，自己做判断。Agent 做判断，展示结果。一些研究者会想覆盖分类结果、调整阈值、或探索其他解读方式。

我们通过让每一步可追溯来缓解这个问题——报告会解释跑了哪些检验、为什么、数字说明了什么。但 MVP 不提供"高级模式"让用户手动覆盖。这个后面再做。

我们也放弃了广度，换取深度。原始 PRD 覆盖 16 种范式，横跨焦虑、抑郁、学习记忆、社会行为、恐惧、运动功能。Agent MVP 覆盖 5 个焦虑范式。这是刻意的取舍——窄而精的体验优于宽而平庸的体验。

## 总结

| | 原始 PRD | Agent 方案 |
|---|---|---|
| 核心隐喻 | 给专家的工具 | 给所有人的助手 |
| 用户假设 | 知道要找什么 | 知道跑了什么实验、哪些是对照组 |
| 主要输出 | 交互式仪表盘 | 组间比较分析报告 + 问答 |
| 专业知识 | 在文档中 | 编码在结构化知识库中（指标定义、统计方法选择、混杂因素检查） |
| LLM 角色 | 未明确 | 仅报告生成 + 问答 |
| 统计分析 | 用户选择检验 | 系统按规则自动选择检验并做组间比较 |
| MVP 范围 | 3 个范式（仅可视化） | 5 个范式（完整流水线） |
| 竞争壁垒 | 功能丰富度 | 领域知识编码 |
| 工程风险 | 高（大量前端工作） | 中等（分析引擎 + 知识库） |

原始 PRD 问的是：怎么让研究者更好地**看到**他们的数据？

我们问的是另一个问题：怎么**告诉**研究者他们的数据意味着什么？
