### 逐组件说明

#### 计算层

**ECS (云服务器)**

- **是什么**：虚拟服务器，裸金属级别的 Linux 机器。
- **在 EthoInsight 的角色**：运行 4 个 Docker 容器（nginx / frontend / gateway / langgraph），是 Agent 分析流水线的实际执行环境。
- **为什么需要**：Agent runtime 必须有 Linux 环境跑 Python 子进程（code-executor 的沙盒执行）、Docker 容器、文件系统挂载。Serverless 函数做不到这些。
- **不用的后果**：自建机房服务器，需要管理硬件、网络、UPS，运维成本远超云服务器。

**SAE (Serverless 应用引擎)**

- **是什么**：免运维的容器托管平台。提交镜像即可运行，无需管理 ECS 实例。
- **在 EthoInsight 的角色**：v0.1 后替代 ECS，托管 Gateway + LangGraph 容器，按请求自动伸缩。
- **为什么需要**：ECS 需要手动打安全补丁、磁盘扩容、配置监控、配置自动重启。SAE 把这些全免掉——日志自动集成 SLS、镜像更新自动滚动发布、异常自动重建。v0.1 10 人时不迫切，v1.0 100 人时必需。
- **不用的后果**：继续 ECS Docker Compose，但运维工作随用户量线性增长（磁盘满、OOM、安全漏洞、手动重启）。

#### 数据层

**RDS PostgreSQL (云数据库)**

- **是什么**：托管的 PostgreSQL 15，自带每日自动备份、监控、读写分离。
- **在 EthoInsight 的角色**：存储用户账号、分析线程（threads）、对话历史、训练数据飞轮反馈、LangGraph checkpoint。
- **为什么需要**：Agent 的对话历史和 checkpoint 必须持久化到数据库，不能存容器内（容器重建就丢）。自建 PG 需要自己做备份、HA、参数调优，RDS 全自动。
- **不用的后果**：容器内 SQLite 只能单用户、不能跨容器共享、数据随时可能随容器重建丢失。5-20 早晨 docker daemon 重建容器导致 hot-patch 全丢就是这类问题的缩影。

**Redis (云缓存)**

- **是什么**：内存 KV 数据库，微秒级读写。
- **在 EthoInsight 的角色**：Rate-limit 计数器（防 API 滥用）、session 缓存（减少 RDS 查询）、验证码 TTL 存储、多用户并发锁。
- **为什么需要**：v0.1 10 人时可以用 PG 临时顶替或用内存字典。50 人以上必须有独立的缓存层——PG 扛不住高频 rate-limit 查询，一次分析会话可能触发几十次 API 调用检查。
- **不用的后果**：Rate-limit 走 PG → RDS CPU 飙升；无 session 缓存 → 每次请求都查 PG → 延迟增加；无并发锁 → 共享资源可能出现竞态条件。

**OSS (对象存储)**

- **是什么**：海量文件存储，S3 兼容接口，99.9999999999% 数据持久性。
- **在 EthoInsight 的角色**：存储研究员上传的 EthoVision 原始数据（.txt / .xlsx）、Agent 生成的分析产物（图表 PNG/SVG、报告 MD/PDF）。
- **为什么需要**：ECS 本地磁盘不持久——实例释放、系统盘故障、误删都可能丢数据。研究员的分析结果本身就是高价值资产（一次实验可能几千元成本），不能丢在本地盘。
- **不用的后果**：数据在 ECS 本地盘，需要自己写备份脚本、定期 tar 打包、手动上传到别处。磁盘满了要手动清理，删错文件无法恢复。

#### CI/CD 层

**ACR (容器镜像服务)**

- **是什么**：Docker 镜像仓库，存储构建产物。
- **在 EthoInsight 的角色**：存放 CI 流水线构建出的 gateway / langgraph / frontend 三个镜像，ECS/SAE 从 ACR 拉取部署。
- **为什么需要**：CI/CD 的核心是"构建一次，到处部署"。没有镜像仓库，代码只能在构建的那台机器上跑（当前手动的 `docker build && docker cp` 模式），换个机器就没了，代码和运行环境脱节。
- **不用的后果**：5-19/5-20 的惨案重现——hot-patch 在容器里，docker daemon 重建容器就全丢。每次部署依赖"人能记住改了什么"，不可重复、不可追溯。

**云效 Flow (CI 流水线)**

- **是什么**：托管 CI 服务，监听 GitHub PR merge 事件，自动执行构建、测试、推送镜像。
- **在 EthoInsight 的角色**：PR merge 到 main → Flow 自动 docker build 三个镜像 → push 到 ACR。所有构建日志可追溯，失败自动通知。
- **为什么需要**：没有 CI，部署靠人手动 run `docker build && docker push`——步骤遗漏、构建环境不一致（你本机 vs 我本机 vs CI 机器）、谁构建的不知道、构建日志找不到。
- **不用的后果**：部署靠"某一台机器上某人手动敲的命令"，无法追溯、无法回滚、无法确保构建环境干净。团队协作时每个新人都要配一遍本地构建环境。

#### 网络与安全

**NAT 网关**

- **是什么**：让 VPC 内没有公网 IP 的资源访问互联网（拉包、更新、调 API）。
- **在 EthoInsight 的角色**：ECS/SAE 内的容器需要 `apt-get` / `pip install` / `npm install` / 拉 GitHub 代码 / 调外部 LLM API，但没有公网 IP 时出不去。NAT 提供统一出口。
- **为什么需要**：不给 ECS 配公网 IP 提升安全性（减少攻击面），但容器仍要访问外部依赖。NAT 是最小成本的方案。
- **不用的后果**：每台 ECS 都要绑公网 IP，安全组暴露面增大；或者容器内无法安装任何外部依赖（等同于断网）。

**WAF (Web 应用防火墙)**

- **是什么**：七层防护，拦截 SQL 注入、XSS、CSRF、CC 攻击、恶意爬虫。
- **在 EthoInsight 的角色**：挡在 DCDN/SAE 前面，过滤所有 HTTP 请求。EthoInsight 有文件上传入口、用户注册、LLM 对话输入——这些都是攻击面。
- **为什么需要**：v0.1 内部 10 人时敏感度低。v1.0 公网开放后，任何暴露在公网的服务都会被扫描——自动化攻击工具几小时内就能发现开放的 Web 端口并尝试已知漏洞。WAF 是第一道防线。
- **不用的后果**：公网暴露的 Web 服务等于在攻击者面前裸奔。AI 应用特有的 prompt injection 攻击（通过用户输入操纵 Agent 行为）如果叠加传统 Web 漏洞，后果从数据泄露到计算资源盗用都可能有。WAF 至少扛住 OWASP Top 10。

**KMS (密钥管理服务)**

- **是什么**：云原生密钥托管，加密存储密码/API Key/Token，支持自动轮转。
- **在 EthoInsight 的角色**：托管 DATABASE_URL（含数据库密码）、百炼 API Key、ACR pull token、JWT secret——所有这些敏感字符串不落盘。
- **为什么需要**：当前 `.env` 文件存明文密码，ECS 被攻破或仓库误上传（`git add .` 把 .env 带上）就直接泄露。KMS 让密码只存在于内存中，应用启动时通过 SDK 获取。
- **不用的后果**：明文密码下，任何能读文件系统的人/进程都能拿到数据库密码和 API Key。对于面向外部用户的服务，这意味着一次入侵可能连带泄露所有用户数据。

**SMS (短信服务)**

- **是什么**：发送短信验证码、通知。
- **在 EthoInsight 的角色**：用户注册时的手机验证码、重大告警通知。
- **为什么需要**：v0.1 内部用户用邮箱注册即可。v1.0 面向外部研究员时，手机短信是标配——邮箱注册有垃圾注册风险，短信验证码能大幅降低虚假账号。
- **不用的后果**：仅有邮箱注册，垃圾账号风险高；管理员无法通过短信收到线上故障告警。

#### 可观测性

**SLS (日志服务)**

- **是什么**：集中式日志平台，支持全文检索、SQL 分析、可视化。
- **在 EthoInsight 的角色**：聚合 4 个容器 + Agent 子代理的所有日志，按 traceId 串联一次完整分析会话。
- **为什么需要**：当前 `docker logs` 看日志，容器重建日志消失；多容器日志分散，追一个问题要翻 4 个日志源；无法按"某次分析会话"过滤关联日志。SLS 让"用户说分析失败了，看下是什么原因"能几秒内定位。
- **不用的后果**：线上问题排查靠 `docker logs | grep ERROR`，日志散、不持久、不可检索，用户反馈问题后无法追溯当时的错误上下文。

**ARMS (应用实时监控)**

- **是什么**：APM 全链路追踪，调用链拓扑、慢查询分析、错误率告警。
- **在 EthoInsight 的角色**：监控 Gateway API 响应时间、LangGraph 子代理执行耗时、RDS 慢查询、LLM API 调用延迟。一次分析从"收到用户消息"到"报告生成完毕"每个环节的耗时一目了然。
- **为什么需要**：Agent 分析链路长（lead → code-executor → data-analyst → report-writer），任一环节变慢或失败都需要快速知道是哪个子代理卡了、是不是 LLM 超时了、还是数据库慢了。没有 APM，定位一次慢会话可能需要逐段翻日志。
- **不用的后果**："最近分析变慢了"——无法判断是 LLM API 延迟、数据库慢查询、还是子代理循环调用 tool。排查靠猜。

#### AI 推理

**百炼 (模型服务平台)**

- **是什么**：阿里云的 LLM API 聚合网关，统一调用 Qwen / DeepSeek / Llama 等模型。
- **在 EthoInsight 的角色**：EthoInsight 的"大脑"——所有 Agent 推理（lead agent 调度、code-executor 生成代码、data-analyst 解读结果、report-writer 写报告）都通过百炼调用 deepseek-v4-pro。
- **为什么需要**：直接调模型厂商 API 需要分别对接、分别计费、分别管理 API Key。百炼统一入口，一个 API Key 调用多个模型，按量付费，有完整的调用审计和 token 统计。
- **不用的后果**：每家模型厂商单独签约、单独充值、API Key 散落各处。切换模型时要改代码中的 endpoint。没有统一的 token 消耗统计。

#### CDN / 加速

**DCDN (全站加速)**

- **是什么**：CDN + 动态加速，前端 JS/CSS 缓存在边缘节点，动态 API 请求走阿里云骨干网加速。
- **在 EthoInsight 的角色**：Next.js 编译出的 JS bundle 可能 2-3MB，研究员在全国各地打开页面时，静态资源从最近的 CDN 节点返回，首屏从 5 秒降到 1 秒内。
- **为什么需要**：v0.1 10 人且都在北京时不迫切。v1.0 用户分布全国时，每次打开页面都从北京 ECS 拉 3MB JS 是糟糕的体验。DCDN 的边际成本极低（流量费几毛钱/GB），但对用户体验的改善巨大。
- **不用的后果**：外地研究员打开页面白屏 5-10 秒，第一印象差。对于 SaaS 产品，首屏加载时间是用户留存的关键指标。
