# EthoInsight 微调小模型方案 — 设计文档

## Context

EthoInsight 当前使用 GLM-4/5（通用大模型）+ prompt engineering 驱动所有 LLM 任务。统计方法选择 ~70% 由 `statistics.py` 硬编码 if-else 处理，~30% 依赖 prompt 注入（实验设计识别、边界情况、报告生成）。

**问题**：
1. if-else 硬编码无法覆盖边界情况（正态性阈值附近、小样本、零膨胀、嵌套设计）
2. Prompt engineering 在生产环境中不够稳定和定制化
3. 依赖通用大模型 API，成本高、不可控
4. 行为学是垂直领域，通用模型缺乏深度领域能力

**目标**：用微调小模型替代通用大模型，实现更准确、更定制化、可本地部署的行为学分析 agent。

---

## 架构：两层 + RAG

```
用户上传数据 + 描述实验
        │
        ▼
┌──────────────────────────┐
│ 第一层：确定性规则引擎      │  ← statistics.py（增强版）
│  • 正态性检验 (Shapiro)    │     100% 可审计、可解释
│  • 方差齐性 (Levene)       │     处理有明确规则的决策
│  • 基础检验选择             │
│  • 效应量计算              │
└────────┬─────────────────┘
         │
         ▼
┌──────────────────────────┐
│ 第二层：微调小模型          │  ← Qwen2.5-7B / Qwen3-8B (LoRA)
│  • 实验设计类型识别         │     一个模型承担所有 LLM 任务
│  • 边界情况方法推荐         │     替代 GLM-4/5 API
│  • 结果解读               │
│  • 报告生成 (APA, 中英双语) │
│  • 领域 Q&A              │
│  • Tool calling           │──→ 调用规则引擎 + 调用 noldus-kb
└──────────────────────────┘
                                    │
                              ┌─────┴─────┐
                              │ noldus-kb  │  ← RAG 检索
                              │ 6200+ 论文 │     知识随时更新
                              └───────────┘
```

**设计原则**：
- 模型学**能力**（怎么分析、怎么写报告）— 通过微调
- 模型访问**知识**（具体文献、参考值）— 通过 RAG
- 不把 6200+ 论文训练进模型（过时、不可追溯、小模型容量有限）

## 基座模型选择

**约束**：强 agent/tool-calling 能力 + 消费级 GPU（RTX 4090 24GB）部署

| 模型 | 参数量 | Tool Calling | 24GB VRAM | 中文 | 推荐度 |
|------|-------|-------------|-----------|------|--------|
| Qwen2.5-7B | 7B | 原生支持，BFCL 排名靠前 | Q4 ~5GB ✅ | 优秀 | ⭐⭐⭐ |
| Qwen3-8B | 8B | 最新，MCP 原生支持 | Q4 ~6GB ✅ | 优秀 | ⭐⭐⭐ |
| GLM-4-9B | 9B | 原生支持 | Q4 ~7GB ✅ | 优秀 | ⭐⭐ |

**推荐 Qwen2.5-7B 或 Qwen3-8B**：tool calling 能力最强，微调工具链最成熟（LLaMA-Factory / Swift），LoRA 训练只需 ~16GB VRAM。

## 训练策略：SFT → DPO 两阶段

### 阶段一：SFT（监督微调）— 教模型"会做"

**四类训练数据**：

| 能力 | 输入 | 输出 | 数据量 |
|------|------|------|--------|
| 设计识别 | 用户自然语言描述 | `{design_type, paradigm, n_groups}` | ~500 条 |
| 方法推荐 | 数据特征 JSON | `{method, reason, alternatives}` | ~300 条 |
| 结果解读 | 统计结果 JSON | 领域解读文本 | ~500 条 |
| 报告生成 | 完整分析输出 | APA 格式 Results 段落 | ~300 条 |
| Tool calling | 用户提问 | 正确的 function call | ~200 条 |

**总计 ~1800 条**，LoRA 微调 7B 模型的合理量级。

### 阶段二：DPO（直接偏好优化）— 教模型"做得好"

用 (输入, 好输出, 差输出) 三元组训练偏好：
- 报告措辞：专业版 vs 泛泛版
- 方法选择：专家推荐 vs 模型原始推荐
- 解读深度：有洞察的 vs 浅层的

**~300-500 对**，来自专家标注。

## 数据构建：三批递进

### 第一批：合成 + 转化（~800 条，1-2 周）

无需专家，从现有资源自动/半自动构建：

| 来源 | 转化方式 | 产出 |
|------|---------|------|
| 统计方法文档 (370 行) | 每个决策规则 → Q&A 对 | ~80 条 |
| SKILL.md 决策树 | 每条路径 → 设计识别样本 | ~50 条 |
| pytest 测试用例 (32 个) | 输入输出直接转为训练数据 | ~32 条 |
| 分析模板 (shoaling.py) | 执行记录端到端样本 | ~10 条 |
| noldus-kb 论文 Methods 段 | 提取方法选择案例 | ~130 条 |
| Claude/GPT-4 合成变体 | 13 范式 × 4 设计 × 20 变体 | ~500 条 |

### 第二批：专家标注（~500 条，周 3-6）

模型上线内部试用后，收集真实交互：
- 记录 (用户输入, 模型输出)
- 专家审核并修正
- 修正数据用于增量 SFT

### 第三批：DPO 偏好数据（~300 对，周 6-8）

从专家标注中提取"好 vs 差"对：
- 专家修正版 = preferred
- 模型原始版 = rejected
- 用于 DPO 训练

### 持续：RLHF 飞轮

```
用户使用 → 收集数据 → 专家标注 → 训练 → 模型更好 → 更多用户
```

## 规则引擎增强（第一层）

在现有 `statistics.py` 基础上补充边界情况：

| 增强项 | 当前状态 | 改进 |
|--------|---------|------|
| 小样本正态性 | α=0.05 一刀切 | n<8 时降低置信度，建议非参数 |
| 零膨胀检测 | 未处理 | 新增 zero_inflation_check() |
| 嵌套设计 | 未处理 | 检测笼/窝效应，提示需混合模型 |
| 多重比较策略 | 只有 Tukey/pairwise | 新增 Holm、FDR 选项 |
| 决策日志 | 无 | 输出每步决策理由（可审计） |

## 执行优先级

**微调和 Phase 0c 并行推进**。去掉 GLM-4/5 API 依赖是刚需（成本可控、数据不出境、本地部署）。

- **工作流 A（Phase 0c）**：run_paradigm_analysis 工具 + E2E 测试 + 内部试用
- **工作流 B（微调）**：数据构建 → SFT → 评估 → DPO → 集成替换

两条线并行，Phase 0c 产出的真实用户交互数据同时喂入微调的训练数据飞轮。

## 验证方式

1. **规则引擎**：扩展 pytest 覆盖新增边界情况，目标 >50 个测试
2. **SFT 效果**：在保留测试集上评估四类能力的准确率
3. **端到端**：用 5 个范式的真实数据跑完整分析流程，对比微调模型 vs 现有 prompt engineering 的输出质量
4. **Tool calling**：验证模型能正确调用规则引擎和 noldus-kb
