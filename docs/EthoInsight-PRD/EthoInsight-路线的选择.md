# 我们究竟是要用Result？还是用Data
> 为防止歧义，下文将软件中的 Statistics Results/统计数据称为`Result`， Raw Data/原始数据称为`Raw`。

## 这些都是什么？
想要弄清`Result`和`Raw`的关系，我们需要捋清软件的分析流程。

一切都开始于采集。采集环节后，除了时间戳以外，我们会得到追踪点的坐标（中心点 或 鼻尖-中心点-尾根 或 色标）、身体长度、身体面积、像素点变化等基础信息。所有的指标和结果都是基于这些基础信息计算而来的。

### Result
在进入`Result`的计算之前，数据会先通过采集环节的轨迹平滑设置进行过滤。

而后根据用户选择的因变量，进行各种计算。
- 根据坐标计算移动距离、速度、航向、等
- 结合分析区设置中的分区信息，根据坐标计算被试动物所在区域
- 根据投影面积计算Mobility，根据像素点变化计算Activity
- 结合身体长度、投影面积、坐标等计算精细行为
- 其它更多的计算组合

这些指标，除了本身的计算方法以外，有的还需要或可以键入其它用户指定的参数，例如：
- 分析区的进入和退出阈值
- 移动 / 活动状态 / 多条件等参数需要用户指定的状态阈值
- 计算条件指定

这些数据还会通过数据选择配置中的要求，进行分类成组或截取，最终呈现给用户。

### Raw
`Raw`和`Result`最大的区别就是数据格式。`Result`是汇总的指标结果，而`Raw`是时序数据，即所有的结果不是汇总的，而是按照每个采样点一行排列的。

除了基础信息以外，`Raw`还会根据分析参数配置中选择的因变量，增加对应参数列。如果在数据选择配置中选择了分组，在列尾也会予以显示。

另外，`Raw`还会在文件的前40行上下输出Meta信息，包含实验的观察区设置名，实验控制设置名，检测设置名。

## 我们该用什么？
在EthoInsight立项之初，团队大部分人倾向使用`Result`作为输入进行分析。这样做的好处是显而易见的——**不重复计算**。这也是经理一开始提到的，*不去做EthoVision做过的事情。*

```Text
可是这样真的好吗？真的有利于后续发展吗？真的能够出洞察出Insight吗？
```
纵观EthoVision的后分析工具，从T3D到EntoLab，再到客户自制的DanioFlow——用的都是`Raw`，无一例外。

*为什么都选择Raw？*

我的回答是，**`Result`是带着镣铐跳舞**——如果真的想去做洞察，拓展更多可能，必然，也只能用`Raw`。原因如下：

1. **高度依赖EthoVision设置**。`Result`的结果是由EthoVision的数据选择配置和分析参数配置决定的。如果用户没有选对应的指标，就无法输出相关结果，也没有方法可以进行补充计算。
2. **低拓展性**。同上一条，由于没有原始数据，所有可以计算的内容被局限在EthoVision自带的因变量中。而所谓`“Insight/洞察”`，需要的不止是EthoVision内建因变量。如果一上来绑死`Result`，会极大的削弱拓展性。
3. **提升兼容性**。按照项目设想，EthoInsight最终会兼容并接入其它第三方工具。我们不能指望要求所有的第三方工具按照同样的方式去输出指标，但是各家的`Raw`格式是类似的——时间戳、坐标、区域——这些是可以通用的。
4. **避免相乘地狱**。架设EthoVision中有3个数据选择配置，5种分析参数配置，`Result`会有 3×5=15 种表格。问题是这15种无法批量生成，需要用户自己一下一下手点输出——这一点很慢，很让人疲惫。而`Raw`仅需选择数据选择配置。

当然，这一切并不是完全没有代价的。最明显的，这和经理一开始希望的——*不做EthoVision干的事情*方向不一致。但如果最终目的是**做出目标中的EthoInsight，需要使用`Raw`，哪怕会多一些工作**。
