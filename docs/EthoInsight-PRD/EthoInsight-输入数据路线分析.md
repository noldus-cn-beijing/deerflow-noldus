# EthoInsight 输入数据路线分析

> 为防止歧义，下文将软件中的 Statistics Results/统计数据称为`Result`， Raw Data/原始数据称为`Raw`。
> 本文档为 `EthoInsight-路线的选择.md` 的扩展分析版本，在原文论证基础上补充了格式细节、MVP 影响评估和落地建议。

---

## 一、两种数据格式是什么？

EthoVision 的分析流程：采集 → 轨迹平滑 → 指标计算 → 数据选择/分组 → 输出。Result 和 Raw 是这条流程的两种输出形态。

### 1.1 采集层：一切数据的源头

采集环节后，除了时间戳以外，我们会得到追踪点的坐标（中心点 或 鼻尖-中心点-尾根 或 色标）、身体长度、身体面积、像素点变化等基础信息。所有的指标和结果都是基于这些基础信息计算而来的。

### 1.2 Result — 汇总指标数据

**格式特征：每只被试动物一行，列为计算后的汇总统计量。**

在进入 Result 的计算之前，数据会先通过采集环节的轨迹平滑设置进行过滤。而后根据用户在 EthoVision 中选择的因变量，进行各种计算：
- 根据坐标计算移动距离、速度、航向等
- 结合分析区设置中的分区信息，根据坐标计算被试动物所在区域
- 根据投影面积计算 Mobility，根据像素点变化计算 Activity
- 结合身体长度、投影面积、坐标等计算精细行为
- 其它更多的计算组合

这些指标除了本身的计算方法以外，有的还需要或可以键入其它用户指定的参数，例如：
- 分析区的进入和退出阈值
- 移动 / 活动状态 / 多条件等参数需要用户指定的状态阈值
- 计算条件指定

这些数据还会通过数据选择配置中的要求，进行分类成组或截取，最终呈现给用户。

**Result 格式示例（EPM 范式）：**

```csv
Subject,openArmTime,openArmEntries,closedArmTime,totalArmEntries,centerTime
Mouse1,18.5,25.3,195.2,22,45.3
Mouse2,16.8,22.1,210.5,19,52.1
Mouse3,20.2,28.4,185.3,24,38.9
```

特征：
- 每行 = 一只动物
- 每列 = 一个已计算好的汇总指标
- 列名由用户在 EthoVision 中选择的因变量决定
- **不可逆**：汇总后无法还原时序信息

### 1.3 Raw — 时序采样数据

**格式特征：每个采样点一行，包含逐帧的坐标、面积、区域等基础信息。**

Raw 和 Result 最大的区别就是数据格式。Result 是汇总的指标结果，而 Raw 是时序数据，即所有的结果不是汇总的，而是按照每个采样点一行排列的。

除了基础信息以外，Raw 还会根据分析参数配置中选择的因变量，增加对应参数列。如果在数据选择配置中选择了分组，在列尾也会予以显示。

另外，Raw 还会在文件的前 40 行上下输出 Meta 信息，包含实验的观察区设置名，实验控制设置名，检测设置名。

**Raw 格式结构示意：**

```text
[Meta 区域，约前 40 行]
# Arena setting: EPM_Standard
# Trial control: 5min_free_exploration
# Detection: nose_center_tailbase
# ...

[时序数据区]
Time,X_center,Y_center,X_nose,Y_nose,X_tail,Y_tail,Body_area,Zone, ...
0.000,250.3,180.5,255.1,172.3,248.0,188.2,1205.3,Closed Arm South
0.040,250.1,180.8,254.8,171.9,248.3,188.5,1203.7,Closed Arm South
0.080,249.8,181.2,254.2,171.2,248.7,189.0,1201.1,Closed Arm South
...
```

特征：
- 每行 = 一个采样点（通常 25 fps，5 分钟实验 ≈ 7,500 行/只动物）
- 列 = 坐标、面积、区域、像素变化等底层信息
- **可计算**：可从时序数据重新计算任意汇总指标
- 包含 Meta 头部信息（实验设置参数）

### 1.4 格式对比总览

| 维度 | Result | Raw |
|------|--------|-----|
| **粒度** | 每只动物一行 | 每个采样点一行 |
| **数据量** | 小（行数 = 动物数） | 大（行数 = 采样点 × 动物数） |
| **内容** | 已计算的汇总指标 | 坐标、面积、区域等底层信息 |
| **可逆性** | 不可逆（无法还原时序） | 可逆（可重新计算汇总指标） |
| **依赖设置** | 高度依赖用户在 EV 中的因变量选择 | 仅依赖采集设置，与分析设置解耦 |
| **Meta 信息** | 无 | 包含约 40 行实验设置元数据 |
| **导出必需项** | 必须在 EV 中选择对应因变量 | 仅需完成采集和基本检测 |

## 二、为什么推荐 Raw？

在 EthoInsight 立项之初，团队大部分人倾向使用 Result 作为输入进行分析。这样做的好处是显而易见的——**不重复计算**。这也是经理一开始提到的，*不去做 EthoVision 过过的事情。*

> 可是这样真的好吗？真的有利于后续发展吗？真的能够出洞察出 Insight 吗？

纵观 EthoVision 的后分析工具，从 T3D 到 EntoLab，再到客户自制的 DanioFlow——用的都是 Raw，无一例外。

*为什么都选择 Raw？*

**Result 是带着镣铐跳舞**——如果真的想去做洞察，拓展更多可能，必然，也只能用 Raw。

### 2.1 理由一：摆脱 EthoVision 设置依赖

Result 的结果是由 EthoVision 的数据选择配置和分析参数配置决定的。如果用户没有选对应的指标，就无法输出相关结果，也没有方法可以进行补充计算。

**具体场景：**
- 用户在 EthoVision 中只选了"开臂时间"和"闭合臂时间"，没有选"总进臂次数"→ Result 中缺少运动混淆检查所需的列，EthoInsight 无法判断焦虑指标是否被运动抑制干扰
- 用户使用默认阈值计算区域进出，但事后想调整进入/退出阈值 → Result 已按原阈值固化，无法重算
- 用户忘记在 EthoVision 中添加特定因变量 → 分析报告中直接缺少该指标

用 Raw 则不存在这个问题：所有指标从时序数据重新计算，不依赖用户在 EthoVision 中做了哪些选择。

### 2.2 理由二：拓展性

由于没有原始数据，Result 可计算的内容被局限在 EthoVision 自带的因变量中。而所谓 `Insight/洞察`，需要的不止是 EthoVision 内建因变量。如果一上来绑死 Result，会极大的削弱拓展性。

**用 Raw 才能实现的拓展能力：**

| 拓展方向 | 具体功能 | 为何 Result 做不到 |
|----------|---------|-------------------|
| **自定义指标** | 用户定义新指标（如"开臂首次进入的路径曲率"） | Result 不含轨迹坐标 |
| **时间窗口分析** | 按任意时间段切片分析（如前 2 分钟 vs 后 3 分钟） | Result 是全程汇总，无时间分辨率 |
| **动态行为模式** | 识别"犹豫-折返"序列、停留时间分布模式 | 需要 timestamp + zone 的逐帧序列 |
| **轨迹分析** | 轨迹热图、路径效率、运动模式聚类 | 需要逐帧 X/Y 坐标 |
| **阈值重算** | 用不同进出阈值重新划分区域停留 | Result 中区域判定已固化 |
| **跨范式个体档案** | 同一个体在不同范式中的行为对比 | 需要可统一计算的底层指标 |

### 2.3 理由三：兼容性与生态

按照项目设想，EthoInsight 最终会兼容并接入其它第三方工具。我们不能指望要求所有的第三方工具按照同样的方式去输出指标，但是各家的 Raw 格式是类似的——时间戳、坐标、区域——这些是可以通用的。

**第三方工具数据格式共性：**

| 通用字段 | 说明 | 出现于 |
|----------|------|--------|
| Timestamp | 采样时间戳 | 所有工具 |
| X, Y 坐标 | 追踪点位置 | 所有工具 |
| Zone/Area | 当前所在区域 | 大多数工具 |
| Body length / Area | 身体长度或面积 | 部分工具 |
| Pixel change | 像素变化量（用于活动性计算） | 部分工具 |

各家的 Raw 格式虽有差异，但核心字段（时间戳 + 坐标 + 区域）几乎是一致的。构建一套通用的 Raw 解析适配层，远比适配各家的 Result 汇总格式现实。

### 2.4 代价与取舍

当然，这一切并不是完全没有代价。最明显的，这和经理一开始希望的——*不做 EthoVision 干的事情*方向不一致。但如果最终目的是**做出目标中的 EthoInsight，需要使用 Raw，哪怕会多一些工作**。

| 选择 | 收益 | 代价 |
|------|------|------|
| **选 Raw** | 拓展性强、兼容性好、不依赖 EV 设置 | 需自行实现指标计算（与 EV 重复）、开发工作量更大 |
| **选 Result** | 开发快、不重复计算 | 功能被 EV 因变量锁死、无法拓展、兼容性差 |
| **混合方案** | MVP 可先用 Result 快速验证，后续迁移至 Raw | 迁移成本、两套逻辑并存的技术债 |

## 三、对 MVP 6 种范式的影响评估

本节逐一分析 Result 和 Raw 路线对每种 MVP 范式的具体影响。

### 3.1 焦虑样行为（4 种）

**EPM — 高架十字迷宫**

核心指标：开臂时间%、开臂进入%、总进臂次数、中心区时间

| 路线 | 能否支持 | 说明 |
|------|---------|------|
| Result | 部分支持 | 能做组间统计检验和效应量，但无法做运动混淆检查（如果用户没导出 `totalArmEntries`），无法做时间段分析 |
| Raw | 完全支持 | 从坐标重新计算所有区域指标，可按任意时间窗口切片，可做轨迹热图 |

**OF — 旷场实验**

核心指标：中心时间%、中心距离%、总距离、直立次数、中心进入次数

| 路线 | 能否支持 | 说明 |
|------|---------|------|
| Result | 基本支持 | OF 指标相对标准，用户一般会全选。但如果缺少 `totalDistance` 则无法做运动混淆检查 |
| Raw | 完全支持 | 从坐标计算距离/速度，结合区域定义计算中心/周边比例 |

**O-Maze — O 迷宫**

核心指标：开放区时间%、犹豫次数、总圈数、方向偏好

| 路线 | 能否支持 | 说明 |
|------|---------|------|
| Result | 有风险 | "犹豫次数"和"方向偏好"是非标准指标，用户很可能没有在 EV 中选择导出 |
| Raw | 完全支持 | 从坐标序列可计算犹豫事件（进入开放区又立即退回）和运动方向 |

**LDB — 明暗箱**

核心指标：明箱时间%、穿梭次数、首次进入潜伏期、明箱距离

| 路线 | 能否支持 | 说明 |
|------|---------|------|
| Result | 基本支持 | 明暗箱指标较简单，潜伏期在 Result 中通常可导出 |
| Raw | 完全支持 | 可从时间戳 + 区域精确计算首次进入潜伏期，可做穿梭模式分析 |

### 3.2 抑郁样行为（2 种）

**TST — 悬尾实验**

核心指标：累计不动时间、不动潜伏期

| 路线 | 能否支持 | 说明 |
|------|---------|------|
| Result | 基本支持 | 不动时间是标准导出项。但无法做不动事件的时段分析（如前 2 分钟 vs 后 4 分钟） |
| Raw | 完全支持 | 从像素变化 / Mobility 时序数据识别不动事件，可做精细时段分析 |

**FST — 强迫游泳**

核心指标：累计不动时间、不动潜伏期

| 路线 | 能否支持 | 说明 |
|------|---------|------|
| Result | 基本支持 | 同 TST。但 FST 的行为细分（攀爬、游泳、不动）在 Result 中可能仅保留"不动时间" |
| Raw | 完全支持 | 从体面积变化和坐标波动区分攀爬/游泳/不动三种行为状态 |

### 3.3 影响总览矩阵

| 范式 | Result 可行性 | Raw 必要性 | 关键风险 |
|------|--------------|-----------|----------|
| EPM | 中 | 高 | 缺 `totalArmEntries` 则无法做运动混淆检查 |
| OF | 中高 | 中 | 指标相对标准，Result 基本够用 |
| O-Maze | 低 | 高 | "犹豫次数"等非标准指标可能缺失 |
| LDB | 中高 | 中 | 指标简单，Result 通常可覆盖 |
| TST | 中 | 中高 | 无法做精细时段分析 |
| FST | 中 | 高 | 行为状态细分需要时序数据 |

## 四、PRD 对齐与落地建议

### 4.1 与 PRD v2 的对齐检查

| PRD 需求 | Result 路线能否满足 | Raw 路线能否满足 |
|----------|-------------------|-----------------|
| US-1.1：AI 自动识别范式类型 | 可以（基于列名模式匹配） | 可以（基于 Meta 信息和坐标结构） |
| US-1.2：数据质量检查 | 可以（缺失值、异常值检测） | 可以（采样点覆盖率、坐标连续性检查） |
| US-2.1：全自动统计检验 | 可以（对已有列做检验） | 可以（先计算指标，再做检验） |
| US-2.3：基于文献阈值的分级 | 可以（对已有指标做分级） | 可以（计算指标后做分级） |
| US-3.3：多轮追问对话 | 受限（"能看看轨迹图吗？"无法实现） | 完全支持（可生成任意可视化） |
| 第二阶段：跨范式分析 | 不支持（缺少可对比的底层指标） | 支持（统一从坐标计算） |
| 第二阶段：轨迹热图 | 不支持 | 支持 |
| 第三阶段：行为模式聚类 | 不支持 | 支持 |
| 第三阶段：异常检测 | 受限 | 完全支持 |

### 4.2 建议路线：分阶段混合策略

考虑到开发资源和时间约束，建议采用分阶段混合策略：

**Phase 1（MVP，前 3 个月）：Result 优先，Raw 预留**

- 主输入格式为 Result，快速实现 PRD 中的 US-1.1 ~ US-4.8
- 在数据导入模块（模块 A）中预留 Raw 解析适配层接口
- 明确告知用户"为获得完整分析，请在 EthoVision 中导出以下列：[范式对应列名清单]"

**Phase 2（扩展期，3-6 个月）：Raw 迁移**

- 实现 Raw 数据的完整解析和指标重算引擎
- 逐步将各范式从 Result 切换到 Raw 输入
- 新增的时间窗口分析、轨迹可视化等 Raw 独有能力在此阶段上线
- 保留 Result 输入作为降级兼容

**Phase 3+（智能化期，6 个月+）：Raw 为主**

- Raw 成为主要输入格式
- Result 作为简易快速分析的可选降级输入
- 第三方数据兼容层基于 Raw 格式构建

### 4.3 MVP 阶段的 Result 列名规范

如果 MVP 阶段使用 Result 路线，需要明确要求用户在 EthoVision 中导出的最小列名集合：

**焦虑样行为（4 种）：**

| 范式 | 必需列 | 运动混淆检查列 |
|------|--------|---------------|
| EPM | `openArmTime`, `openArmEntries`, `closedArmTime`, `centerTime` | `totalArmEntries` |
| OF | `centerTimeRatio`, `centerDistanceRatio`, `totalDistance` | `totalDistance`, `averageSpeed` |
| O-Maze | `openTimeRatio`, `openEntriesRatio` | `totalLaps` |
| LDB | `lightTimeRatio`, `transitions`, `firstEntryLatency` | `transitions` |

**抑郁样行为（2 种）：**

| 范式 | 必需列 | 运动混淆检查列 |
|------|--------|---------------|
| TST | `immobilityTime`, `immobilityLatency` | `immobilityTime`（自身即为活动性指标） |
| FST | `immobilityTime`, `immobilityLatency` | `immobilityTime`（自身即为活动性指标） |

> MVP 阶段需要在导入流程中检测这些必需列是否存在，缺失时提示用户回到 EthoVision 补充导出。

## 五、总结

| 维度 | Result | Raw |
|------|--------|-----|
| **开发速度** | 快 | 慢（需实现指标计算引擎） |
| **MVP 可行性** | 中高风险（3 种范式可能缺关键列） | 高（不依赖用户 EV 设置） |
| **拓展天花板** | 低（被 EV 因变量锁死） | 高（可计算任意自定义指标） |
| **第三方兼容** | 困难（各家 Result 格式不统一） | 可行（核心字段通用） |
| **与 PRD 愿景对齐** | 部分对齐（MVP 够用，扩展受限） | 完全对齐（支持所有阶段目标） |

**结论：** Raw 是实现 EthoInsight 完整愿景的必要选择。考虑到开发资源约束，MVP 可先以 Result 快速验证核心流程，但架构上必须为 Raw 迁移做好准备。长期来看，Raw 是唯一能支撑"从数据到洞察"这一产品定位的数据路线。
